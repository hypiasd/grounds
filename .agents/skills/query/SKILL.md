# query

## 何时用（触发）
- 用户问："之前学的 X 是怎样的"、"复习一下 Y"、"对比 A 和 B"。
- 用户就仓库里已有笔记的内容提问。
- 区分 query vs. learn：query 回答已有笔记覆盖的内容；learn 处理完全陌生的新概念。

## 目标（完成时仓库应处于的状态）
- 基于仓库笔记给出**带引用**的答案；内容不足时明说并建议 learn；不凭空编造。
- query 默认不写文件，只有用户要求沉淀时才走 capture。

## 流程
1. **定位**：
   - **第一遍（快速扫描）**：列 `wiki/`，只读每篇笔记的 frontmatter（`title`、`tags`、`summary`、`topic`）。这一步不加载正文，只用 summaries 判断哪些笔记和用户的问题相关。默认只扫 `wiki/`；如用户问题明确指向论文（"那篇论文/那篇 paper"），可补扫 `paper/`，引用时需标注"来自论文笔记"。
     - 推荐用 Bash 提取 frontmatter（避免加载正文）：`awk '/^---$/{flag++; next} flag==1' wiki/<topic>/<note>.md`
     - 或直接 `head -15 <file>`（frontmatter 一般在前 15 行内）
     - 多文件批量扫：`for f in wiki/**/*.md; do echo "=== $f ==="; head -15 "$f"; done`
   - **第二遍（精准加载）**：只读扫描命中的笔记正文。命中超过 5 篇时，按 summary 相关度排序后只加载前 5 篇正文；其余在答案末尾以"另有相关笔记：..."列出链接供用户自选。如果扫描发现没有相关笔记，或加载后判定笔记不足以回答，直接进入边界判断，不继续加载更多正文。
2. **综合策略**（按场景选择）：
   - **单篇笔记足够**：直接基于该笔记作答，引用具体段落。
   - **跨多篇笔记**：先列举相关笔记的要点（各一两句），再对比或串联，最后给出综合结论。对比类问题的特殊处理：
     - A、B 都在仓库 → 按"列举要点 → 对比 → 结论"作答，对比段可用表格或并列分点
     - A 在、B 不在 → 答 A 部分，明说 B 无笔记，问用户是否走 learn 补 B 或走外部搜索
     - A、B 都不在 → 不假设用户记错，先列仓库内最相关的笔记（如有），明说无 A/B 直接笔记，建议 learn
   - **笔记不够但你知道**：明确说"仓库里没有相关笔记，但我可以说说已知的——以下内容未沉淀，建议后续走 learn 存下来"。**自评"知道"时谨慎——如果不确定自己记得对，归入"完全不知道"分支建议 learn，避免把幻觉当已知输出。**
   - **完全不知道**：直接说"仓库没有，我也不确定"，建议 走 learn（可带外部资料）。
3. **引用粒度**：引用到具体笔记文件（如 `[Dropout](deep-learning/dropout.md)`），如果有多个相关笔记，需逐一标注。**query 答案中的链接路径相对 `wiki/` 目录**（如 `deep-learning/dropout.md`），便于用户在仓库中定位。query 输出的链接不强制要求关系说明（conventions.md 约束的是笔记文件），但建议在链接后用一句话点明该笔记与用户问题的关系，方便用户判断是否点开。
4. **校验（必做）**：对答案中每个引用路径执行 `test -f wiki/<topic>/<file>.md && echo OK`（Bash），未返回 OK 的链接必须从答案中删除或改为"（仓库中暂无相关笔记）"。不要引用不存在的笔记。
5. **沉淀/修复建议**：根据情况建议不同 skill：
   - 用户获得新理解/新角度 → 建议走 **capture**
   - 发现已有笔记矛盾/过时/断链 → 建议跑 **lint**（不要用 capture 修笔记，capture 是增量沉淀不是修订）
   - 同时存在两者 → 先建议 lint 修旧，再 capture 存新
   - 答案本身是有价值的新综合视角 → 建议走 capture；query 本身不强制写文件。

## 何时进入边界判断

- 第一遍扫描无相关笔记，或第二遍加载后判定笔记不足以回答用户问题 → 进入下方"边界判断模板"。

## 何时在边界判断内触发外部搜索

- 用户问题包含"最新/现在/今年"等时间敏感词
- 用户明确要求"顺便查一下网上"
- 仓库完全无笔记且 agent 也不确定（"完全不知道"分支）
- 对比类问题中一方在仓库、另一方完全不在

若以上都不满足，跳过"实时搜索补充"段，直接给"建议 learn"。

## 边界判断模板

当仓库内容不足以回答时（按上方"何时进入边界判断"判定），使用以下标准回复结构：

```
## 仓库内相关内容
- [笔记A](<topic>/a.md) — 涉及了 X，但没有深入 Y（你的问题）
- [笔记B](<topic>/b.md) — 提到了 Z，但角度不同

## 实时搜索补充（仅在满足"何时触发外部搜索"条件时启用；否则跳过本段）
（如需联网搜索，用 Exa：`mcporter call 'exa.web_search_exa(query: "...", numResults: 5)'`）
- Exa 搜索：[结果摘要]
- 其他平台（小红书/知乎等）：用 `opencli <platform> search "..." -f yaml`

## 建议
- 满意的话可以走 capture 沉淀搜索发现
- 想深入理解可以走 learn（可带上搜到的文章链接）
```

## query vs learn 边界（部分覆盖场景）

- 用户问的概念在仓库有"父概念/相邻概念"笔记，但没有该概念本身 → 走 **learn**（新建笔记），learn 流程会处理与相邻笔记的互链
- 用户问的是已有笔记的某个子角度 → 走 **query**，答完后若子角度值得独立成篇则建议 capture
- 不确定时优先 query（轻量），用户嫌浅再升级 learn

## 质量示例

**好的 query 回答**：
> 根据 [Attention Mechanism](deep-learning/attention-mechanism.md) 的笔记，核心思想是让模型在处理每个 token 时动态关注输入序列中不同位置的信息。笔记里用"查字典"做了类比：Query 是你要查的词，Key 是字典的索引，Value 是释义。常见误区是以为 Attention 只在 Transformer 里有——实际上它最早来自 RNN 的 Seq2Seq 模型。
>
> ——有引用、有类比、有误区、定位具体。

**差的 query 回答**：
> 注意力机制是深度学习中的一种技术，用于让模型关注输入的不同部分。它在 NLP 中很常用。
>
> ——没有引用、没有深度、读完和没读一样。

## 工具使用

- `Read`：读笔记正文（第二遍精准加载）
- `Bash`：第一遍扫描时提取 frontmatter（避免加载正文）、校验链接目标存在性（`test -f <path>`）
- 不使用 Write/Edit（query 默认不写文件）

## Gotchas（真实踩过的坑）

- **答案必须可溯源到仓库笔记**，禁止凭空发挥——这是该 skill 存在的意义。如果答案和笔记对不上，用户会产生"笔记白写了"的信任危机。
- **仓库内容不足时明说**，并建议用户 走 learn（可带外部资料），不要硬编。曾有 agent 假装引用笔记但笔记里根本没那个内容。
- **发现过时 / 矛盾 → 提示用户跑 lint**，不要自己默默改笔记。因为你可能在 query 上下文里看不到全貌，静默修改可能引入新矛盾。
- **引用前必须确认链接目标存在**：用 `test -f wiki/<topic>/<file>.md && echo OK` 校验，未通过的链接删除。用户点击一个断链的体验比没有链接更差。
- **外部搜索不是默认动作**：仅在"何时触发外部搜索"条件满足时启用，避免对每个边界判断都联网搜（慢且常多余）。
- **多笔记命中时不要硬全部加载**：超过 5 篇时按 summary 相关度排序后只加载前 5 篇，其余列出链接即可。
