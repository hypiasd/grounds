# capture

## 何时用（触发）
- 用户说："把这轮对话整理一下"、"记下来"、"沉淀一下"。
- 一轮对话结束后，用户希望把学到的东西固化进仓库。
- 注意：learn skill **只讲解、不沉淀**。learn 讲完后如需固化收获，由用户手动触发 capture（用 `$capture` 或 '@capture'）——不要假设 learn 已经或应该自己保存。

## 目标（完成时仓库应处于的状态）
- 对话中所有有价值的原子洞察已各归其位——每个洞察要么更新到已有笔记，要么新建笔记。
- 所有涉及的 `index.md` 已同步，一次 `git commit && git push` 覆盖全部变更。
- **不**另存原始对话原文。

## 核心原则

**一次对话 → 多个原子洞察 → 各归其位。** 一段有价值的对话通常会触及多个概念——不要强行把所有内容塞进一篇笔记。

Zettelkasten 的做法：先蒸馏（提取所有独立的洞察），再合成（每个洞察归位）。对应已有笔记的，做增量追加；没有对应笔记的，新建。

## 流程

### 第一步：蒸馏 + 切分方案

回顾当前对话，提取**所有**原子洞察。每个洞察应满足：
- 是一个独立的概念性发现（不是一个事实碎片）
- 能在 2-3 句话内说清楚
- 要么深化了已有笔记，要么值得新建一篇
- **对话中涉及代码的必须把代码记下来**——代码是概念最精确的表达。不要只记「用 triton.jit 装饰器定义一个 kernel」而不给代码片段

然后将洞察组织成**笔记切分方案**，列出清单给用户审核。格式：

```
本轮对话 → N 篇笔记：

1. 新建 `wiki/<topic>/<note>.md` — [概念名]
   内容：[洞察一句话]

2. 更新 `wiki/<topic>/<note>.md` — [概念名]
   内容：[洞察一句话]，加进「常见误区」板块

3. 不用存 — [洞察一句话，说明为什么跳过]
```

**切分方案必须给用户审核，不可跳过。** 用户有权：
- 合并："2 和 3 合成一篇"
- 拆分："1 太泛了，拆成两篇"
- 改归属："1 放在 X 主题下更合适"
- 删除："3 不用存"
- 确认："方案没问题，执行"

**在呈现切分方案的同时，必须一并询问用户「需不需要搜索面经」**（用 `ask_followup_question` 给出「需要 / 不需要」两个选项，与切分方案的审核合并为一次确认，不要等到第三步才突然问）。把面经需求与切分方案一起定下来——用户说「不需要」则后续第三步整步跳过；说「需要」则第三步照常执行。用户的意愿以当次为准，不要沿用上一轮对话的偏好。

切分方案是 capture 最重要的决策节点——几篇笔记、各自归属——这个判断不应由 agent 独断。

**用户选择违反原子性时的追问**：当用户提出"合成一篇"、"全放一起"等操作，且涉及的内容明显覆盖 3 个以上独立概念时，不要直接执行——先追问一次。格式：

```
> 这几个是独立概念（[列举 2-3 个]），合成一篇会让日后 query 定位变难 —— 比如查「vector 扩容」必须加载整篇几百行的 STL 全景笔记，而不是直接命中一个原子页。确定合吗？
```

追问的原则：
- **只问一次**，不要反复纠缠。用户确认后直接执行，不再劝。
- **追问里必须点出具体的概念名**（不能笼统说"违反原子性"），让用户感知到代价。
- **适用范围**：覆盖 3+ 个独立概念的合并才触发；合并 2 个或纯增量更新（如"把这个误区加进已有笔记"）不需要追问。
- **如果用户先问了"有什么代价"，再回答**——不要先斩后奏。



### 第一步（附）：opencli 环境检查

面经搜索主力依赖 `opencli`，每次 capture 第一步就做连通性检查，不要等到第三步才发现连不上：

```bash
opencli doctor 2>&1 || echo "BROKEN"
```

- 返回正常（含 `Connectivity: connected` / `Extension connected` 等字样）→ OpenCLI 三源 + WebSearch 中英文全部可用，完整执行。
- 返回 `BROKEN` 或明显未连接字样（如 `Connectivity: disconnected`、`not connected`）→ **不停止流程**。告知用户"opencli 未连接，面经搜索将仅用 WebSearch，覆盖度会降低。如需完整搜索请打开 Chrome 并启用 OpenCLI 扩展"——然后继续切分和归位，到第三步时仅走 WebSearch + WebFetch 路径（中英文两路都发）。

### 第二步：归位


用户确认切分方案后，对每个洞察执行：

**匹配已有笔记**：
- 扫 `wiki/` 找相关笔记。读笔记的 frontmatter（`title`、`summary`、`tags`）快速判断是否相关，确认相关后读正文。
- 如果洞察是对已有笔记的深化/补充 → **更新模式**：将新内容融入已有结构（类比加进「直觉/类比」，误区加进「常见误区」，关联加进「关联」），刷新 `updated` 日期。
- 如果洞察是全新的、独立的概念 → **新建模式**：读 `.agents/conventions.md`，按模板新建 `<note>.md`。

**更新模式的关键规则**（来自 create-wiki-llm 的 additive surgical edits）：
- 只做增量追加，不全篇重写。已有的好内容不动。
- 新内容融入对应板块，不创建重复板块。
- 如果新洞察显著改变了核心理解，同步更新 `summary`。

**新建模式**：
- 按 conventions.md 模板写（概念名标题、tags、summary 等）。
- 判断归属 `wiki/<topic>/`，沿用或新建主题（新建须带 `index.md`）。

### 第三步（前置）：是否搜索面经（沿用第一步的答复）

**面经的询问已在第一步呈现切分方案时一并完成，本步默认沿用第一步的答复，不重复询问。** 只有在第一步因故未问到的情况下，才在此补问：

- 第一步用户说「需要」→ 继续执行下方第三步（opencli 三源 + WebSearch 两路搜索、深入阅读、筛选写入）。
- 第一步用户说「不需要」→ **整步跳过**，不跑 `opencli doctor`、不搜任何源，直接进入第四步批量写入。
- 仅当第一步遗漏了面经询问时，才用 `ask_followup_question` 在此补问「需要 / 不需要」，等用户明确答复后再决定。

### 第三步：面经补充

**仅当用户在「第三步（前置）」确认需要时才执行本步。** 归位完成后，对涉及的每个概念，搜索网络上真实的面试题，补充到笔记中。

**操作**（使用 OpenCLI 直接调用）：

> **为什么要加面经**：learn 的检验是 AI 出题，面经是业界实际在问的题——两者角度不同。面经告诉你「这个知识在现实中怎么被考核」，往往是笔记里写了的定义和类比都覆盖不到的角度。

**第一步：搜索**

面经搜索**分两路并行**：OpenCLI 三源（中文面经主力）+ WebSearch（通用网页补漏，覆盖面经社区、技术博客、英文资源）。

##### 第一路：OpenCLI 三源（中文面经主力）

对每个涉及的笔记，**必须搜完以下三个源，一个都不能跳过。** 禁止搜到一个满意的就停。

- **小红书** → `opencli xiaohongshu search "[中文概念名] 面试" --limit 5 -f yaml`
- **知乎** → `opencli zhihu search "[中文概念名] 面试" --limit 5 -f yaml`
- **牛客** → `opencli nowcoder search "[中文概念名] 面试" --limit 5 -f yaml`
每个源单独搜，各自返回结果。不要用通用网页搜索去搜 `site:zhihu.com`——会被反爬拦死，必须走 OpenCLI。

##### 第二路：WebSearch + WebFetch（通用网页补漏，尽力而为）

> **重要**：WebSearch 和 WebFetch 不是所有 agent 平台都有效。发出即可——无结果、报错、超时都**不阻塞流程**，静默跳过。面经搜索的硬性要求只有 OpenCLI 三源。

OpenCLI 的三源覆盖中文面经社区，但可能遗漏：
- 英文面试资源（LeetCode Discuss、Reddit r/cscareerquestions、Glassdoor）
- 技术博客中的面试题整理
- 小众社区和论坛讨论

**WebSearch** 用中文和英文分别搜索，并行发出（有就发，没有就跳过）：

- `WebSearch("[中文概念名] 面试题")`（中文）
- `WebSearch("[English concept name] interview questions")`（英文）

**WebFetch** 对搜索结果中的高价值链接深入读取正文。同样有就发，没有就跳过。

不可用时的影响：面经覆盖度降低（缺少英文社区和技术博客视角），但 OpenCLI 三源已覆盖中文面经主力渠道，仍可产出有效结果。

##### 并行化

OpenCLI 三源（必须） + WebSearch 中英文（尽力） = 最多 5 个搜索并行发出。深入阅读阶段同样并行——同一平台内多篇文章、多个 WebFetch 链接可同时发起。跨阶段（搜索 → 深入阅读 → 筛选）存在依赖时才串行。

```
搜索阶段（OpenCLI 三源必发，WebSearch 尽力）    深入阅读（并行）          筛选写入（串行）
┌─ opencli zhihu ───────┐
├─ opencli xhs ─────────┤               ┌─ download/answer ─┐
├─ opencli nowcoder ────┤ → 汇总结果 →  ├─ note ─────────────┤
├─ WebSearch 中文 ──────┤ (尽力)        ├─ nowcoder 正文 ─────┤ → 去重 → 筛选 → 写入
└─ WebSearch 英文 ──────┘ (尽力)        └─ WebFetch × N ────┘ (尽力)
```

**WebSearch/WebFetch 不可用时**：OpenCLI 三源是硬要求，WebSearch 是软补充。无结果或报错时静默跳过，不阻塞、不重试、不报告用户。面经质量受影响但不影响流程完整性。

**第二步：深入阅读**

搜索结果的标题和摘要不够——**必须点进具体内容读完整原文**，从原文中提取 Q&A，不是从搜索摘要里猜。

- **知乎**：搜索结果中两类链接需要不同的命令：
  - `zhihu.com/question/xxx/answer/xxx`（问答）→ `opencli zhihu answer-detail <answer_id> -f json`
  - `zhuanlan.zhihu.com/p/xxx`（专栏文章）→ `opencli zhihu download --url "<完整 URL>" -f json`，返回 JSON 中 `saved` 字段指向导出的本地 Markdown 文件（如 `zhihu-articles/<标题>/<标题>.md`），**用 `cat` 直接读该文件获取正文**。专栏文章通常内容最长、质量最高，是知乎面经的主要载体，跳过等于浪费最大信息源。
  - **铁律**：任何 `zhihu.com` / `zhuanlan.zhihu.com` 链接**都只能用 opencli 读取，严禁用 `WebFetch`**——WebFetch 会被知乎登录墙拦截、返回"知乎欢迎你"空页，浪费一次调用还拿不到内容。即便该 URL 来自 WebSearch 结果，也先判域名，是知乎就走 opencli，绝不用 WebFetch。

  **读取完毕后必须清理**：`zhihu-articles/` 是下载缓存目录，读完正文后立即删除整个目录——不留中间产物污染仓库，也避免下次 capture 读到旧缓存。
  - 不要用 `answer-detail` 去读专栏 URL——会直接报错且丢失内容。
- **小红书**：对搜索结果中相关度高的笔记，用 `opencli xiaohongshu note "<完整 URL>" -f json` 读正文。注意小红书笔记如果是纯图片，正文可能为空——跳过
- **牛客**：对搜索结果中相关度高的帖子，OpenCLI 返回的内容字段可能已包含完整正文，直接提取；如截断则跳过
- **WebSearch 结果**：用 `WebFetch` 读取高价值链接（面经帖、面试题集合页、技术博客中的面试题整理）。一次 3-5 个链接即可——目标是补 OpenCLI 三源覆盖不到的角度，不是逐条穷举。WebFetch 返回的 markdown 正文直接从中提取 Q&A。但**若链接域名是 `zhihu.com` 或 `zhuanlan.zhihu.com`，禁止用 WebFetch**——改走 `opencli zhihu download --url "<URL>" -f json`（专栏）或 `opencli zhihu answer-detail <id> -f json`（问答），见上方「知乎」一节。

**第三步：筛选与去重**

从所有源（OpenCLI 三源 + WebSearch）的完整内容中筛选 5-10 个题目。**先读笔记已有的「面试常见问题」段落（如有），建立已收录题目的指纹（题干归一化：去标点/空格/问号、转小写后比对），新题干与已收录重合度 >70% 时跳过**——避免同一题反复追加导致笔记膨胀。

多源间也要去重：同一题在多个源都出现时，只记一次，来源字段列多个（如 `*来源：知乎 • @xxx • <链接>；WebSearch 英文 • <链接>*`），代表此题覆盖度高、值得收录。

筛选标准：
- 笔记里讲过的概念，面试中会被怎么问？——优先收录
- 笔记里写了常见误区，面试中是否真的有对应的坑？——高分题
- 面试题本身揭示了笔记没有覆盖的角度——最有价值，收录后考虑是否需要补充笔记内容
**第四步：写入**

将 Q&A 追加到笔记末尾，格式：

```markdown
## 面试常见问题

- **Q**: [面试题原文或转述]
  **A**: [参考答案——用自己的话，不是复制粘贴]
  *来源：[知乎/小红书/牛客/WebSearch] • [作者名] • [链接]*

- **Q**: ...
```

- 如果已有的笔记已包含面试问题板块 → 新的 Q&A 追加进去，不覆盖。

**搜索原则**：
- OpenCLI 三源（小红书/知乎/牛客）是硬要求——必须全搜，不能跳过
- WebSearch + WebFetch 是软补充——能发就发，不可用时静默跳过
- 中文面经是主力（用户大概率面中文岗位），英文 WebSearch 覆盖英文社区和论文方法
- 不用搜太偏门的来源——面试题的价值在于代表性，不在多
- 如果搜索结果质量差或找不到，跳过，不要硬凑

### 第四步：批量写入

所有洞察写入完成后：

1. **检查完整性**：每个涉及的 `index.md` 是否已同步（新建的笔记是否已加入索引？更新的笔记如果核心内容变了，index.md 的介绍是否同步？）。
2. **批量校验**：`wc -l` 确认每个涉及的文件非空；确认所有新增/修改的笔记 frontmatter 完整（`title`、`topic`、`tags`、`summary`、`created`/`updated`）；**frontmatter 的字段值若含 `:`、`#`、`"`、`[`、`{` 等 YAML 特殊字符（最常见的是 `summary` 里写了 `::` 作用域符或 `key: value` 式冒号），整个值必须用双引号包裹**（如 `summary: "…:: 作用域解析…"`），否则 Quartz `npx quartz build` 会报 `bad indentation of a mapping entry` 解析失败；检查所有新增相对链接目标存在。
3. **一次提交**：所有变更一个 commit，不拆多个。先 `git status` 确认仅本批笔记 + 涉及的 `index.md` 改动；显式 `git add` 每个涉及的文件（不要 `git add -A`，避免误带其他未完成改动）：
   ```
   git add wiki/<topic>/<note1>.md wiki/<topic>/<note2>.md wiki/<topic>/index.md && \
   git commit -m "capture grounds: 沉淀对话笔记 <YYYY-MM-DDTHH-MM>" && git push
   ```
   如果所有洞察都落在同一个 topic，commit message 可以用 `capture <topic>: ...`；否则统一用 `grounds`。

## 什么是"值得沉淀"

**值得沉淀**：
- 对话中产生了新的理解（不是搬运已知事实）
- 有具体的关键结论或待验证的假设
- 填补了仓库现有知识的空白，或深化了已有笔记
- 用户明确说"这个很重要，记下来"

**不值得沉淀**（在清单中主动标注"不用存"）：
- 日常闲聊、问候、流程性对话
- 和已有笔记高度重复的内容
- 只有一句话的简单结论，不值得单独成篇
- agent 不确定、无法验证的内容

## 质量示例

**好的 capture**：
```
提取到 3 个洞察：

1. BatchNorm 的 intuition 可以更新——之前只说了"减均值除方差"，
   但对话中你想到的"标准化考试分数"类比更直观 → 更新 batch-norm.md
2. LayerNorm 和 BatchNorm 的区别在对话中讲清楚了，但还没有笔记 → 新建 layer-norm.md
3. Dropout 和 BatchNorm 的互斥关系是个新发现 → 更新 dropout.md 和 batch-norm.md 的「关联」部分

3 个洞察，涉及 2 篇更新 + 1 篇新建，1 个 commit。
```

**差的 capture**：
```
把整个对话塞进一篇叫"关于归一化和正则化的讨论"的笔记里——150 行，
包含了 BatchNorm、LayerNorm、Dropout、互斥关系、面试题...什么都有一点，什么都找不到。
```

## Gotchas（真实踩过的坑）

- **不要全量搬运对话**：蒸馏是减法——先列出所有可能的洞察，再筛掉重复的、太浅的、不确定的。曾有 agent 把整段对话逐条拆成多篇笔记，导致 wiki 膨胀。
- **写入后必须校验非空**：并行写文件曾静默丢失内容（报成功但 0 行）。批量写入时每个文件都要独立校验。
- **更新的笔记也要完整校验**：更新已有笔记时，容易漏掉 frontmatter 字段（尤其是 `updated` 日期和 `summary` 是否需要刷新）。更新后必须重新检查所有必填字段。
- **index.md 同步**：新建笔记必须加入对应主题的 index.md；更新笔记如果核心内容变了也要同步。批量操作时容易漏掉这一步。
- **不另存完整对话原文**：agent 无法可靠保真地逐字存 transcript，对话精华应在笔记中提炼，而非全量搬运。
- **对话中的代码必须记下来**：不能只记描述——"用了 triton.jit 装饰器"不够，要记完整的代码片段。代码是概念最精确的表达，也是面试和实际应用中最需要的东西。
- **面经必须多源搜完，不能搜到一个满意的就停**：搜到一个源就停——不行。OpenCLI 三源（小红书/知乎/牛客）是硬要求，必须全部搜完。WebSearch + WebFetch 是尽力而为的补充——能发就发，不可用时静默跳过。不同源覆盖的角度不同，所有源搜完、深入阅读后，再汇总筛选。
- **WebSearch/WebFetch 不可用不阻塞流程**：这两个工具不是所有 agent 平台都有效。发出即可，无结果/报错/超时都静默跳过——不重试、不报告用户、不影响后续步骤。
- **capture 和 learn 的边界**：learn 只负责讲解与检验，不做沉淀。learn 讲完后用户想保存收获 → 由用户手动触发 capture。capture 处理所有需要固化进仓库的对话收获（包括 learn 讲解后的内容）。

## 注意
- capture 是**独立** skill，不和 learn 自动绑定；用户掌控节奏。
