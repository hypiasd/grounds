# capture

## 何时用（触发）
- 用户说："把这轮对话整理一下"、"记下来"、"沉淀一下"。
- 一轮对话结束后，用户希望把学到的东西固化进仓库。
- 注意：learn skill 讲解后如果用户说"存一下"，由 learn 自己处理，不需要再走 capture。

## 目标（完成时仓库应处于的状态）
- 对话中所有有价值的原子洞察已各归其位——每个洞察要么更新到已有笔记，要么新建笔记。
- 所有涉及的 `_overview.md` 已同步，一次 `git commit` 覆盖全部变更。
- **不**另存原始对话原文。

## 核心原则

**一次对话 → 多个原子洞察 → 各归其位。** 一段有价值的对话通常会触及多个概念——不要强行把所有内容塞进一篇笔记。

Zettelkasten 的做法：先蒸馏（提取所有独立的洞察），再合成（每个洞察归位）。对应已有笔记的，做增量追加；没有对应笔记的，新建。

## 流程

### 第一步：蒸馏 + 切分方案

回顾当前对话，提取**所有**原子洞察。每个洞察应满足：
- 是一个独立的概念性发现（不是一个事实碎片）
- 能在 2-3 句话内说清楚
- 要么深化了已有笔记，要么值得新建一篇

然后将洞察组织成**笔记切分方案**，列出清单给用户审核。格式：

```
本轮对话 → N 篇笔记：

1. 新建 `wiki/<topic>/<note>.md` — [概念名]
   内容：[洞察一句话]

2. 更新 `wiki/<topic>/<note>.md` — [概念名]
   内容：[洞察一句话]，加进「常见误区」板块

3. 不用存 — [洞察一句话，说明为什么跳过]
```

**切分方案必须给用户审核，不可跳过。** 用户有权：
- 合并："2 和 3 合成一篇"
- 拆分："1 太泛了，拆成两篇"
- 改归属："1 放在 X 主题下更合适"
- 删除："3 不用存"
- 确认："方案没问题，执行"

切分方案是 capture 最重要的决策节点——几篇笔记、各自归属——这个判断不应由 agent 独断。

### 第二步：归位

用户确认切分方案后，对每个洞察执行：

**匹配已有笔记**：
- 扫 `wiki/` 找相关笔记。读笔记的 frontmatter（`title`、`summary`、`tags`）快速判断是否相关，确认相关后读正文。
- 如果洞察是对已有笔记的深化/补充 → **更新模式**：将新内容融入已有结构（类比加进「直觉/类比」，误区加进「常见误区」，关联加进「关联」），刷新 `updated` 日期。
- 如果洞察是全新的、独立的概念 → **新建模式**：读 `.agents/conventions.md`，按模板新建 `<note>.md`。

**更新模式的关键规则**（来自 create-wiki-llm 的 additive surgical edits）：
- 只做增量追加，不全篇重写。已有的好内容不动。
- 新内容融入对应板块，不创建重复板块。
- 如果新洞察显著改变了核心理解，同步更新 `summary`。

**新建模式**：
- 按 conventions.md 模板写（概念名标题、tags、summary 等）。
- 判断归属 `wiki/<topic>/`，沿用或新建主题（新建须带 `_overview.md`）。

### 第三步：面经补充

归位完成后，对涉及的每个概念，搜索网络上真实的面试题，补充到笔记中。

**为什么要加面经**：learn 的检验是 AI 出题，面经是业界实际在问的题——两者角度不同。面经告诉你「这个知识在现实中怎么被考核」，往往是笔记里写了的定义和类比都覆盖不到的角度。

**操作**：
1. 对每个涉及的笔记（新建或大幅更新的），**必须使用 Tavily 搜索** `[概念名] 面试题` 或 `[concept] interview questions`。禁止凭训练数据编造面试题——只有网上搜到真实存在的题才加。
2. 筛选 2-3 个高质量题目——优先选那些需要真理解的，而不是背诵定义就能答的。
3. 将 Q&A 追加到笔记末尾，格式：

```markdown
## 面试常见问题

- **Q**: [面试题原文或转述]
  **A**: [参考答案——用自己的话，不是复制粘贴]
- **Q**: ...
```

4. 如果已有的笔记已包含面试问题板块 → 新的 Q&A 追加进去，不覆盖。

**搜索原则**：
- 优先找中文面经（牛客、知乎、CSDN），因为用户大概率面中文岗位
- 但如果概念本身是英文社区的（如论文方法），也搜英文
- 不用搜太偏门的来源——面试题的价值在于代表性，不在多
- 如果搜索结果质量差或找不到，跳过，不要硬凑

### 第四步：批量写入

所有洞察写入完成后：

1. **检查完整性**：每个涉及的 `_overview.md` 是否已同步（新建的笔记是否已加入索引？更新的笔记如果核心内容变了，_overview 的介绍是否同步？）。
2. **批量校验**：`wc -l` 确认每个涉及的文件非空；确认所有新增/修改的笔记 frontmatter 完整（`title`、`topic`、`tags`、`summary`、`created`/`updated`）；检查所有新增相对链接目标存在。
3. **一次提交**：所有变更一个 commit，不拆多个。
   ```
   git add -A && git commit -m "capture grounds: 沉淀对话笔记 <YYYY-MM-DDTHH-MM>"
   ```
   如果所有洞察都落在同一个 topic，commit message 可以用 `capture <topic>: ...`；否则统一用 `grounds`。

## 什么是"值得沉淀"

**值得沉淀**：
- 对话中产生了新的理解（不是搬运已知事实）
- 有具体的关键结论或待验证的假设
- 填补了仓库现有知识的空白，或深化了已有笔记
- 用户明确说"这个很重要，记下来"

**不值得沉淀**（在清单中主动标注"不用存"）：
- 日常闲聊、问候、流程性对话
- 和已有笔记高度重复的内容
- 只有一句话的简单结论，不值得单独成篇
- agent 不确定、无法验证的内容

## 质量示例

**好的 capture**：
```
提取到 3 个洞察：

1. BatchNorm 的 intuition 可以更新——之前只说了"减均值除方差"，
   但对话中你想到的"标准化考试分数"类比更直观 → 更新 batch-norm.md
2. LayerNorm 和 BatchNorm 的区别在对话中讲清楚了，但还没有笔记 → 新建 layer-norm.md
3. Dropout 和 BatchNorm 的互斥关系是个新发现 → 更新 dropout.md 和 batch-norm.md 的「关联」部分

3 个洞察，涉及 2 篇更新 + 1 篇新建，1 个 commit。
```

**差的 capture**：
```
把整个对话塞进一篇叫"关于归一化和正则化的讨论"的笔记里——150 行，
包含了 BatchNorm、LayerNorm、Dropout、互斥关系、面试题...什么都有一点，什么都找不到。
```

## Gotchas（真实踩过的坑）

- **不要全量搬运对话**：蒸馏是减法——先列出所有可能的洞察，再筛掉重复的、太浅的、不确定的。曾有 agent 把整段对话逐条拆成多篇笔记，导致 wiki 膨胀。
- **写入后必须校验非空**：并行写文件曾静默丢失内容（报成功但 0 行）。批量写入时每个文件都要独立校验。
- **更新的笔记也要完整校验**：更新已有笔记时，容易漏掉 frontmatter 字段（尤其是 `updated` 日期和 `summary` 是否需要刷新）。更新后必须重新检查所有必填字段。
- **_overview.md 同步**：新建笔记必须加入对应主题的 _overview；更新笔记如果核心内容变了也要同步。批量操作时容易漏掉这一步。
- **不另存完整对话原文**：agent 无法可靠保真地逐字存 transcript，对话精华应在笔记中提炼，而非全量搬运。
- **capture 和 learn 的边界**：learn 讲解后用户当场说"存"→ 由 learn 自己处理，无需再走 capture。capture 处理的是 learn 之外的对话收获。

- capture 是**独立** skill，不和 learn 自动绑定；用户掌控节奏。
