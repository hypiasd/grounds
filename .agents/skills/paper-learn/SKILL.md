# paper-learn

## 何时用（触发）

**只接受手动触发**：
- 用户明确说"用 paper-learn 读这篇论文"、"走 paper-learn"
- 用户输入 `$paper-learn <url>` 或 `$paper-learn <本地 PDF 路径>`

**不得自动触发**：即使用户在对话里贴了 arxiv/DOI 链接，没有显式调用本 skill，agent 不得自行启动。可以先问一句"要用 paper-learn 读吗？"，但不能直接开跑。

## 目标（完成时仓库应处于的状态）

- 用户对这篇论文有**结构化理解**：解决什么问题、提什么方法、关键贡献、局限、在领域中的位置。
- 仓库新增一篇 `paper/<topic>/<论文标题>.md`（或更新已有笔记），已 `git commit + push`。
- 源 PDF 存放于 `raw/papers/<arxiv-id 或短标题>.pdf`。

## 核心原则

**以学习者视角为主、批判性读者为辅**。不是 reviewer 在做 Accept/Reject 评审，是学习者吃透一篇论文。

借鉴三个公开论文 skill 的设计：
- [bytedance academic-paper-review](https://github.com/bytedance/deer-flow/blob/main/skills/public/academic-paper-review/SKILL.md)：key claims 提取、6 维方法学评估表、论文类型适配
- [paper-craft-skills/paper-analyzer](https://github.com/zsyggg/paper-craft-skills)：论文-代码对照（搜 GitHub 开源实现）
- CSDN 文献汇报方法论：分章节拆解（背景/问题/方法/实验/结果/创新/局限），关键结论标注论文来源

## 流程

paper-learn 分五个阶段：**准备 → 论文全景 → 分章节深入 → 检验 → 沉淀**。

### 阶段零：准备

1. **获取论文**：
   - 用户提供 arxiv URL → 下载 PDF 到 `raw/papers/<arxiv-id>.pdf`（如 `raw/papers/1706.03762.pdf`）
   - 用户提供 DOI/其他 URL → 下载 PDF，文件名用论文短标题（小写中划线）
   - 用户提供本地 PDF 路径 → 复制到 `raw/papers/`，文件名同上
   - 下载命令示例：
     ```bash
     # arxiv
     curl -L -o raw/papers/1706.03762.pdf "https://arxiv.org/pdf/1706.03762"
     ```

2. **查 paper/ 是否已有该论文笔记**：
   - 按 arxiv_id 或 title 匹配。读 frontmatter 的 `title` 和正文"论文信息"段。
   - **已有笔记** → 必须先读笔记正文，告诉用户："笔记里已有 X 和 Y，但 Z 部分只有骨架——需要从 Z 开始补，还是你有其他方向想深入？"等用户确认。
   - **没有笔记** → 「这是个新论文，我按新建模式来」→ 进入阶段一。

3. **提取 metadata**：标题 / 作者 / venue / 年份 / arxiv_id（从 PDF 或 arxiv API）。

---

### 阶段一：论文全景

**目标**：让用户在 30 秒内知道这篇论文在做什么、在领域中的位置。不是 learn 的"概念地图"，是论文骨架。

**格式**：

```
## <论文标题> 全景

**解决什么问题**：[1-2 句，论文要解决的具体问题]

**提出什么方法**：[1-2 句，核心方法名 + 一句话机制]

**关键贡献**：
1. [贡献 1]
2. [贡献 2]
3. [贡献 3]

**在领域中的位置**：[1-2 句，和已有工作的关系，是开创性/改进/应用/理论突破]

**论文类型**：[Empirical / Theoretical / Survey / Systems / Position]
```

**结束信号**：全景之后给用户选择权——

「以上是这篇论文的全景。你想怎么读？
- 按论文结构逐段读（Abstract → Intro → ... → Conclusion）
- 聚焦某个模块（如只深读 Method，或只看 Experiments）
- 都讲讲」

---

### 阶段二：分章节深入

用户选择后，按论文结构深入讲解。**一次一个章节**——讲透一章再问用户要不要继续。

论文标准结构：`Abstract → Intro → Related Work → Method → Experiments → Discussion → Conclusion`。不是所有论文都有全部章节，按实际结构走。

每个章节的讲解结构：

1. **核心内容**
   - 这一章讲了什么（用自己的话，不是翻译）
   - **公式/图/算法必须写出来**（同 learn 铁律）：不能说"用 attention 计算"而不给公式
   - 关键图表的解读（论文里的 Figure X 说明了什么）

2. **作者论证逻辑**
   - 为什么这么设计（不只是 what，还有 why）
   - 作者怎么说服读者这个方法 work

3. **Key claims 提取**（借鉴 academic-paper-review）
   ```
   Claim 1: [具体声称]
   Evidence: [论文里的证据——哪个实验/图/表/推导]
   Strength: [Strong / Moderate / Weak]
   ```

4. **常见误区**（1-2 个，论文没说但容易踩的坑）
   - 例如：方法适用边界、符号歧义、容易误解的术语

5. **（可选）论文-代码对照**（借鉴 paper-analyzer）
   - 搜 GitHub 是否有官方/第三方开源实现
   - 把方法描述和代码实现对应起来（如"论文式 (3) 对应 `model.py` 第 45 行的 `forward` 方法"）
   - 只在用户选了"深读 Method"或明确要复现时启用，不强制

**篇幅**：一个章节 3-5 个要点，不要展开到子子章节。

一个章节讲完后——

「关于 [章节名]，有什么疑问吗？清楚了的话，检验一下这个章节的理解？还是继续下一章？」

---

### 阶段三：检验

对**当前深入章节**或**整篇论文**进行检验（如果用户已读完整篇）。

**第一层：基础检验（1-2 问）**

从以下论文版问法中选 1-2 个。每次一问——先让用户回答，**用户答完后**再给推荐答案对照。

| # | 问法 | 测什么 |
|---|------|--------|
| 1 | "用你自己的话复述作者的方法，不许抄 Method 段" | 是否内化 vs. 复述原文 |
| 2 | "实验为什么这么设计？换一个 baseline 会怎样？" | 对实验设计的理解 |
| 3 | "这篇论文最大的局限是什么？如果你来改进会怎么做？" | 批判性思考 |
| 4 | "作者的核心 claim 有没有被实验充分支撑？哪个实验最关键？" | claim-evidence 对应 |
| 5 | "这个方法和 [仓库已有笔记里的相关方法] 有什么关联？" | 知识整合 |
| 6 | "如果让你复现，最容易踩的坑是什么？" | 复现可行性 |

**回答扎实**：用自己的话、能说实验设计逻辑、能指出局限、能和已有知识关联。
**回答有漏洞**：只是复述论文原文、说不出实验为什么这么设计、对局限性说"好像没什么问题"。

**第二层：面试官深挖（基础检验扎实时自动触发）**

- **追问方向必须来自用户自己的回答**，顺着那个点往下挖。
- **1-2 层即可，触边就收。**
- 追问方向候选：
  - 和同期工作对比（论文没引的）
  - 复现可行性（哪些细节论文没说清）
  - 扩展到其他场景（方法能迁移吗）

---

### 阶段四：补漏与继续

- 检验有漏洞 → 针对性补充讲解。
- 面试官深挖触到边界 → 确认边界位置，简短补充。

补完后问用户：

「[章节名/这篇论文] 先学到这。还想继续深挖其他章节，还是沉淀？」

用户选择继续 → 回到阶段二。用户说沉淀 → 进入阶段五。

---

### 阶段五：沉淀

1. **判断是否值得沉淀**：对照下方"值得沉淀"标准。
2. 若值得 → 问用户"要沉淀成笔记吗？"。用户确认后：
   - **新建模式**：列 `paper/` 判断主题（沿用或新建 `<topic>/`，新建须带 `_overview.md`）→ 新建 `<论文标题>.md` → 更新 `_overview.md`。
   - **更新模式**：读已有笔记 → 将新内容融入现有结构 → 刷新 `updated` 日期。
   - **关键原则**：一篇论文一个 md 文件，文件名即论文标题。两次读同一篇论文是同一篇笔记的两次迭代。
3. **校验（必做）**：`wc -l <note.md>` 确认非空；确认 frontmatter 完整；`git status` 确认改动符合预期。
4. **提交**：`git add -A && git commit -m "paper-learn <topic>: <一句话>"`，然后 `git push`。

## paper 笔记模板

frontmatter 用 conventions 基础字段（不扩展），论文专属信息放正文：

```markdown
---
title: <论文完整标题>
topic: <topic 名，如 llm / cv / rl>
tags: [tag1, tag2, tag3]
summary: <2-3 句话概括核心贡献，给 query 扫描用>
created: 2026-07-17
updated: 2026-07-17
source: ../../raw/papers/<arxiv-id 或短标题>.pdf
---

# <论文标题>

## TL;DR

<2-3 句：解决什么问题 + 核心方法 + 关键结论。读完这一段就知道这篇论文在说什么。>

## 论文信息

- **作者**：[作者列表，第一作者 et al. 即可]
- **Venue**：[会议/期刊名]  **年份**：[年份]  **arxiv**：[arxiv_id 或 N/A]
- **源文件**：[PDF](../../raw/papers/<arxiv-id>.pdf)
- **论文类型**：[Empirical / Theoretical / Survey / Systems / Position]

## 研究背景与动机

- 领域要解决什么问题
- 现有方法的不足
- 本文的动机

## 核心方法

- 方法框架（用自己的话讲，不是翻译 Method 段）
- 关键公式（**必写**，用 LaTeX `$$...$$`）
- 与已有方法的区别

## 实验设计与关键结果

- 数据集 / baseline / 评价指标
- 2-3 个关键结果（**标注来自论文哪个图/表**，如"见原论文 Table 2"）
- 消融实验（如有）

## 创新点与贡献

1. [贡献 1]
2. [贡献 2]
3. [贡献 3]

## 局限与改进方向

- 方法假设
- 实验不足
- 泛化能力
- 可能的改进方向

## 方法论评估（可选）

| 维度 | 评分 | 简评 |
|------|------|------|
| Soundness | /5 | [技术正确性] |
| Novelty | /5 | [新意] |
| Reproducibility | /5 | [复现可行性] |
| Experimental Design | /5 | [实验设计] |
| Statistical Rigor | /5 | [统计严谨性] |
| Scalability | /5 | [可扩展性] |

## 我的批判性思考

- 最关键的一个 claim 是什么？证据够吗？
- 如果复现，最容易踩的坑是什么？
- 和已学知识的关系（可单向引用 wiki 笔记，如"对比 [Attention](../../wiki/llm/attention.md)"）

## 关联

- 相关 wiki 笔记（单向引用，不建反链）：[笔记名](../../wiki/<topic>/<note>.md) — 关系说明
- 同期/后续工作（可附 arxiv 链接）
```

> 结构为建议，非强制。但 TL;DR、核心方法（含公式）、实验设计与关键结果、局限与改进方向四者**至少要有三个**——缺了它们，笔记退化成论文摘要搬运。

---

## 什么是"值得沉淀"

**值得沉淀**：
- 用户读完论文后能用自己的话讲清楚方法
- 有清晰的 key claims 和 evidence 对应
- 填补了仓库论文笔记的空白
- 检验中用户展现出对局限性的批判性思考
- 能写出简洁的论文标题、至少 2 个 tags、让 query 能扫描定位的 summary

**不值得沉淀**（讲完直接结束）：
- 用户只是快速浏览，没有深入
- 只读了 Abstract 就放下
- 和已有论文笔记高度重复（如同系列工作的 v2 版本，增量更新即可，不新建）

---

## Gotchas（真实踩过的坑）

- **不接受语义触发**：用户贴 arxiv 链接但没说"用 paper-learn"→ 不得自动开跑。先问一句"要用 paper-learn 读吗？"。
- **先给论文全景，不要一上来就逐段讲**：让用户先看到骨架，自己决定读法。
- **一次只讲一个章节**：不要同时讲 Method 和 Experiments。讲完一章 → 检验/答疑 → 用户选下一章或沉淀。
- **检验不是可选环节**：每章深入后或整篇读完后必须检验。
- **有公式必须写出来**：不能说"用 attention 计算"而不给公式。
- **沉淀不是默认终点，是用户选择**：补漏后问用户继续还是沉淀。
- **推荐答案在用户回答之后给**：先出题让用户答，答完后再给推荐答案对照。
- **一次只问一个问题**：禁止一次抛出多个问题。
- **不要未经确认就写文件**：仍要问"要沉淀成笔记吗？"。
- **写入后必须校验文件非空**：曾发生并行写文件静默丢失内容。
- **同一篇论文永远只有一篇笔记**：已有笔记 → 更新模式。
- **论文-代码对照是可选的**：不强制每次都搜 GitHub，只在用户深读 Method 或要复现时启用。
- **不要做 Accept/Reject 评审**：本 skill 是学习者视角，不是 reviewer。方法论评估表是可选的反思工具，不是评审结论。
- **讲完忘 commit + push**：commit 之后必须 push，否则换机器看不到。

## 注意

- paper 笔记和 wiki 笔记物理隔离，不参与互链。paper 笔记可单向引用 wiki，但 wiki 不反向链接 paper。
- 主题结构沿用 wiki 的 topic 划分（如 `paper/llm/`、`paper/cv/`），但**不合并不拆分**——一篇论文一个文件，论文不会移动。
- 关联：`AGENTS.md`、`.agents/conventions.md`（wiki 笔记规范，paper 不套用但可参考）
