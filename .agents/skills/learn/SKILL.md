# learn

## 何时用（触发）
- 用户问一个新知识 / 概念 / 原理，或想对已有知识深入某个方向。
- 用户说："讲讲 X"、"X 是什么"、"帮我理解 Y"。
- 用户问的内容**在仓库现有笔记中找不到，或笔记未覆盖用户想深挖的角度**（如果笔记已完全覆盖，走 query）。

## 目标（完成时仓库应处于的状态）
- 用户对该概念有**全局认知**（知道它是什么、为什么存在、有哪些子方向、和什么相关），并能**在自己选择的方向上有深入理解**。
- 若用户认可沉淀：仓库新增/更新一篇 `wiki/<topic>/<note>.md`，对应 `index.md` 已同步，且已 `git commit && git push`。

## 流程

learn 分五个阶段：**查仓库 → 全景概览 → 选方向深入 → 检验 → 沉淀**。全景概览给地图，选方向后才是真正的教学。

### 阶段零：查仓库（讲解前必做）

在开始讲解之前，先扫 `wiki/` 确认两件事。**只看 wiki/，不看 raw/**——raw/ 是原始资料，不是用户学过的证明。

**一、是否已有该概念的笔记：**

- **已有笔记 → 必须先读笔记正文**，了解当前实际覆盖了什么。然后**停下来告诉用户你的判断**：「笔记里已有 X 和 Y，但 Z 部分只有骨架——需要从 Z 开始补，还是你有其他方向想深入？」等用户确认后再进入对应模式。
- **笔记内容丰富，且用户的提问已被充分覆盖** → 告诉用户"笔记里已有 X 和 Y，不过如果关于 Z 的部分想深挖可以继续"。**等用户决定**——禁止自动重定向到 query。
- **没有相关笔记** → 「这是个新概念，我按新建模式来」→ 进入前置知识检查。

**二、前置知识检查（新建模式必做）：**

扫 `wiki/` 判断用户是否具备学这个概念所需的前置知识。**无论扫出什么结果，必须停下来等用户回复——这不是通知，是提问。** 禁止说"前置没问题，我开讲了"然后直接进入讲解。检查结果必须是一个以问号结尾的问题。

- 前置笔记**存在** → 「笔记里有 Y 和 Z 涉及了前置概念 A、B、C——这些你真的掌握了吗？需要简单回顾还是直接开始？」
- 前置笔记**不存在** → 「X 依赖 Y 和 Z——我注意到你的笔记里还没有这些。之前学过吗？还是先从基础开始？」
- 前置笔记**存在但内容浅**（仅骨架）→ 「笔记里有 Y 但是只有定义，看起来还没有深入——需要先补一下 Y 还是你有基础？」

**每种情况都以提问结束，必须等用户回复。** 用户说"学过/没问题/开始吧"就进入讲解；说"没学过/回顾一下"就调整目标。

**三、外部资料（按需）：**

- **用户已手动放入 `raw/wiki/`** → 直接用。
- **用户提供 URL/仓库/视频** → 下载或 clone 到 `raw/wiki/<来源名>/`（文件名小写中划线）。
- **用户只给了一个主题名** → 可用 Exa 搜参考文章（`mcporter call 'exa.web_search_exa(query: "...", numResults: 5)'`），但不强制。

### 阶段一：全景概览

**目标**：让用户看到这个概念的完整地图——它是什么、为什么存在、有哪些子方向、和什么相关。广度优先，每个方向只给 1-2 句，不展开深度。

**格式**：

```
## [概念名] 全景

**是什么**：[1-2 句话，最简单的定义 + 一个直觉类比]

**为什么需要它**：[1-2 句话，它解决了什么问题]

**核心脉络**：
1. **[方向 A]** — [1 句话说明这个方向讲什么]
2. **[方向 B]** — [1 句话说明这个方向讲什么]
3. **[方向 C]** — [1 句话说明这个方向讲什么]
4. **[方向 D]** — [1 句话说明这个方向讲什么]
...（4-8 个方向，覆盖这个概念的主要子话题）

**和其他概念的关系**：[1-2 句话，和用户已学概念的关联]
```

**约束**：
- 每个方向只写一句话——不是一段，不是一篇。这是地图上的标注，不是旅游攻略。
- 方向数量取决于概念本身的复杂度。简单的概念 3-4 个方向，复杂的概念 6-8 个方向。
- 不要在此阶段给公式、代码、误区——那些是深入阶段的材料。
- 不要替用户判断哪些方向「重要」或「值得学」——所有主要方向都列出来，让用户自己选。

**结束信号**：全景概览之后，**必须给用户选择权**——

「以上是 [概念名] 的全景。你想从哪个方向开始深入？可以选一个，也可以说"都讲讲"我逐个来。」

---

### 阶段二：深入讲解

用户选择方向后，对该方向进行深度讲解。**一次只深入一个方向**——讲透一个再问用户要不要继续下一个。

每个方向的讲解结构：

1. **直觉/类比**（1 个，让用户先"感受到"这个方向在说什么）
2. **核心内容**（形式化定义、关键机制、公式——**有公式必须写公式**）
3. **常见误区**（1-2 个，这个方向上最容易踩的坑）
4. **与其他方向的关联**（1 句——这个方向和全景中的其他方向怎么连起来）

**篇幅**：一个方向控制在 3-5 个要点，不要展开到子子方向。如果用户想继续深挖，可以在检验之后选「继续深挖」。

一个方向讲完后，进入答疑——

「关于 [方向名]，有什么疑问吗？清楚了的话，我们检验一下这个方向的理解，然后你可以选择继续深入其他方向，或者沉淀。」

---

### 阶段三：检验

对**当前深入的方向**进行检验。如果用户选了多个方向，每个方向深入后各自检验。

检验分两层：先基础检验确认理解，回答扎实则自动升级到面试官深挖。

**第一层：基础检验（1-2 问）**

从以下六种中选 1-2 个。每次一问——先让用户回答，**用户答完后**再给出推荐答案做对照。

| # | 问法 | 测什么 |
|---|------|--------|
| 1 | "用你自己的话解释 X，不许用术语" | 是否内化 vs. 背定义 |
| 2 | "X 和 Y 的区别？给个具体场景"（Y 选一个容易混淆的相似概念） | 边界辨别 |
| 3 | "什么时候 X 不适用？或者用了反而更差？" | 对局限性的理解 |
| 4 | "给我一个实际会遇到的例子" | 抽象→具象的映射 |
| 5 | "关于 X，初学者最容易踩的坑是什么？你确定你没踩？" | 元认知 |
| 6 | "X 和你学过的 Z 有什么关联？"（Z 必须来自仓库已有笔记） | 知识整合 |

**回答扎实**：用自己的话、有例子、能说边界和局限、能和已有知识关联。
**回答有漏洞**：只是复述定义、给不出例子、对局限性说"好像都可以"。

**第二层：面试官深挖（基础检验扎实时自动触发）**

如果基础检验用户回答扎实，不要满足——你是面试官，你的目标是**找到他理解的边界**。

- **追问方向必须来自用户自己的回答**，顺着那个点往下挖。
- **1-2 层即可，触边就收。**
- **追问方式**：挑战用户自己的类比和表述。比如用户说"Attention 就像查字典"→ 追问"那如果字典里的词条会随着你查的动作实时变化呢？"

---

### 阶段四：补漏与继续

- 检验有漏洞 → 针对性补充讲解。
- 面试官深挖触到边界 → 确认边界位置，简短补充。

补完后问用户：

「[方向名] 先学到这。全景里还有 [列举剩余未学的方向]——你想继续深入哪个？还是都差不多了，先沉淀？」

用户选择新方向 → 回到阶段二（只对该方向深入讲解 + 检验，不需要重走全景概览）。用户说沉淀 → 进入阶段五。

---

### 阶段五：沉淀

1. **判断是否值得沉淀**：对照"好笔记"标准——有类比、有误区、有自己的话。
2. 若值得 → 问用户"要沉淀成笔记吗？"。用户确认后：
   - **新建模式**：列 `wiki/` 判断主题（沿用或新建，新建须带 `index.md`）→ 读 `.agents/conventions.md` → 新建 `<note>.md` → 更新 `index.md`。
   - **更新模式**：读 `.agents/conventions.md` → 读已有笔记 → 将新内容融入现有结构 → 刷新 `updated` 日期 → 更新 `index.md`。
   - **关键原则**：同一概念永远只有一篇笔记。两次学 Dropout 是一篇笔记的两次迭代。
3. **校验（必做）**：`wc -l <note.md>` 确认非空；确认 frontmatter 完整；`git status` 确认改动符合预期；检查笔记内链接目标存在。
4. **提交**：`git add -A && git commit -m "learn <topic>: <一句话>" && git push`。

## 全局交互规则

**每次阶段切换或发起提问前，必须先确认。** 每个确认都要带后续动作，不让用户对着黑箱回答。

- 全景概览结束 → 「你想从哪个方向开始深入？可以选一个，也可以说"都讲讲"。」
- 深入讲解每个方向后 → 「关于 [方向名]，有什么疑问吗？清楚了的话，检验一下？还是想继续深入这个方向的其他细节？」
- 检验每问答完 → 「可以下一问吗？下一问是：[问题方向]。」
- 补漏后 → 「[方向名] 先学到这。全景里还有 [剩余方向]——继续深入哪个？还是先沉淀？」
- 面试官深挖前 → 「基础没问题的话，我顺着你的回答再挖一层，可以吗？」

简化为一条铁律：**每次确认都告诉用户下一步是什么，不让用户盲选。**

## 什么是"值得沉淀"

**值得沉淀**：
- 用户表现出困惑后恍然大悟
- 内容有清晰的直觉类比和常见误区
- 填补了仓库现有知识的空白
- 检验中用户自己的精辟总结或独特理解角度
- 能够写出简洁的概念名标题、至少 2 个 tags、以及一个让 query 能扫描定位的 summary

**不值得沉淀**（讲完直接结束）：
- 用户只是随口一问，对话一笔带过
- 内容太浅（一句话就能说清楚的定义）
- 和已有笔记高度重复

## Gotchas（真实踩过的坑）

- **先给地图，不要一上来就深入一个点**：全景概览是 learn 最重要的阶段——用户不知道有什么可学的，就无法做出选择。跳过全景直接深入，等于替用户决定什么重要。
- **全景要覆盖主要方向，不要漏**：用户没选的方向不代表不重要。如果某个主要方向没列出来，用户永远不会知道它的存在。
- **一个方向讲透再换下一个**：不要同时深入两个方向。讲完一个 → 检验 → 用户选下一个或沉淀。
- **检验不是可选环节**：每个方向深入后必须检验。
- **沉淀不是默认终点，是用户选择**：补漏后展示全景剩余方向让用户选，不是"可以沉淀了吗？"。
- **推荐答案在用户回答之后给**：先出题让用户答，答完后再给推荐答案对照——出题时附答案等于剧透。
- **一次只问一个问题**：禁止一次抛出多个问题。
- **不要未经确认就写文件**：仍要问"要沉淀成笔记吗？"。
- **写入后必须校验文件非空**：曾发生并行写文件静默丢失内容。
- **先查前置知识，不要假设用户懂**：wiki 无基础笔记 → 主动问。
- **同一概念永远只有一篇笔记**：已有笔记 → 更新模式。
- **笔记存在 ≠ 用户掌握**：先读笔记正文。禁止自动重定向到 query。
- **raw/ 不是知识的证明**：只看 wiki/。
- **有公式必须写出来**。
- **阶段零必须等用户回复，禁止直接开讲**。
