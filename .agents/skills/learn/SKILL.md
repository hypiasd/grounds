# learn

## 何时用（触发）
- 用户问一个新知识 / 概念 / 原理，或想对已有知识深入某个方向。
- 用户说："讲讲 X"、"X 是什么"、"帮我理解 Y"。
- 用户问的内容**在仓库现有笔记中找不到，或笔记未覆盖用户想深挖的角度**（如果笔记已完全覆盖，走 query）。

## 目标（完成时仓库应处于的状态）
- 用户已理解该概念（直觉 + 形式化 + 误区都覆盖），**且经过主动检验确认理解扎实**。
- 若用户认可沉淀：仓库新增/更新一篇 `wiki/<topic>/<note>.md`，对应 `_overview.md` 已同步，且已 `git commit`。

## 流程

### 阶段零：查仓库（讲解前必做）

在开始讲解之前，先扫 `wiki/` 确认两件事。**只看 wiki/，不看 raw/**——raw/ 是原始资料，不是用户学过的证明。用户可能下载了一堆论文但一篇都没读。

**一、是否已有该概念的笔记：**

- **已有笔记 → 必须先读笔记正文**，了解当前实际覆盖了什么。然后**停下来告诉用户你的判断**：「笔记里已有 X 和 Y，但 Z 部分只有骨架——需要从 Z 开始补，还是你有其他方向想深入？」等用户确认后再进入对应模式。
- **笔记内容丰富，且用户的提问已被充分覆盖** → 告诉用户"笔记里已有 X 和 Y，不过如果关于 Z 的部分想深挖可以继续"。**等用户决定**——禁止自动重定向到 query。
- **没有相关笔记** → 「这是个新概念，我按新建模式来」→ 进入前置知识检查。

**二、前置知识检查（新建模式必做）：**

扫 `wiki/` 判断用户是否具备学这个概念所需的前置知识。**无论扫出什么结果，必须停下来等用户回复——这不是通知，是提问。** 禁止说"前置没问题，我开讲了"然后直接进入讲解。检查结果必须是一个以问号结尾的问题。

- 前置笔记**存在** → 「笔记里有 Y 和 Z 涉及了前置概念 A、B、C——这些你真的掌握了吗？需要简单回顾还是直接开始？」
- 前置笔记**不存在** → 「X 依赖 Y 和 Z——我注意到你的笔记里还没有这些。之前学过吗？还是先从基础开始？」
- 前置笔记**存在但内容浅**（仅骨架）→ 「笔记里有 Y 但是只有定义，看起来还没有深入——需要先补一下 Y 还是你有基础？」

**每种情况都以提问结束，必须等用户回复。** 用户说"学过/没问题/开始吧"就进入讲解；说"没学过/回顾一下"就调整目标。

**三、外部资料（按需）：**

如果学习这个概念需要外部资料：

- **用户已手动放入 `raw/`** → 直接用，沉淀时通过 `sources` 指回。
- **用户提供 URL/仓库/视频** → 下载或 clone 到 `raw/`（文件名小写中划线），大文件先告知体积。
- **用户只给了一个主题名，没有具体资料** → 可以在讲解开始前用 Tavily 搜一下相关参考文章作为讲解素材。

外部资料为讲解服务——资料到位后再开始讲，不是讲完了才补。

**每次阶段切换或发起提问前，必须先确认。** 每个确认都要带后续动作，不让用户对着黑箱回答。具体来说：

- 讲解结束 → 基于刚讲的内容给出 2-3 个可追问的方向，如："关于 X 我展开讲过数学推导，Y 的实现细节也可以展开，Z 和其他概念的对比也能聊聊——你想深入哪个方向？还是都没问题，直接进入检验？"（不给方向只问"有什么疑问"，等于让人面对空白。）
- 答疑循环中每次解答完 → 同样给方向："这个点讲完了，还想深入 [下一个可展开的方向]，还是进入检验？"
- 基础检验每问答完 → "可以下一问吗？下一问是：[问题方向]。"
- 进入面试官深挖前 → "基础部分没问题的话，我们深挖一层——我会顺着你的回答追问，直到找到你的理解边界。可以吗？"
- 面试官追问答完 → "继续往下挖（更深一层），还是换方向？还是进入补漏？"
- 补漏讲解后 → "这部分补充清楚了吗？清楚了的话，我们可以看看接下来深入的方向，还是先沉淀？"
- 阶段末尾 → 给出 2-3 个具体深入方向 + 沉淀选项。如："你可以继续的方向有——X 的数学推导、Y 的代码实现、Z 和其他概念的对比。先沉淀还是继续深挖某个方向？"

简化为一条铁律：**每次确认都告诉用户下一步是什么，不让用户盲选。**

### 四个阶段：讲解 → 检验 → 补漏 → 沉淀

检验是固定阶段，不需要用户说"考考我"。新建模式和更新模式共享这四个阶段，区别仅在沉淀时是新建文件还是更新已有文件。

### 阶段一：讲解（精简，给最少必要信息）

讲解的目标不是「把你知道的都倒出来」，而是**给到能开始检验的最小信息量**。深度细节在检验和补漏阶段展开——用户答不上来的时候再讲，比一口气全讲完有效得多。

**篇幅约束**：
- 控制在 3-5 个要点以内，不要超过 ~300 字（代码块和公式不计入）
- 1 个直觉类比 + 1 个核心公式（如适用）+ 1-2 个常见误区 —— 够了，停
- 不要列完整 API、不要枚举所有变体、不要给 benchmark 数据、不要展开实现细节
- 如果你发现自己写了表格、目录、分段小标题超过 3 个——你已经在写教程了，删

**内容要求**：
1. 先给**直觉 / 类比**（1 个，最多 2 个）。
2. 给出核心的形式化定义或公式。**有公式必须写公式**。
3. 指出 1-2 个最常见的误区。
4. 不确定处明确说"我不确定"，不要编造。
5. **更新模式额外要求**：讲解时主动关联已有笔记的内容。

**结束信号**：讲完 3-5 个要点后，进入**答疑循环**：

1. 问："这部分有什么疑问吗？"
2. 用户提出疑问 → 针对性解答（只解答用户问的，不趁机扩展新话题）。
3. 解答完后**再次问**："还有其他不清楚的地方吗？"
4. 重复 2-3，直到用户说"没了"、"可以了"、"开始吧"。
5. **用户确认没有问题后，才进入检验。**

用户的困惑在检验之前全部解决掉。带着不理解的东西被考，检验只是徒增挫败感。

### 阶段二：检验（AI 主动发起，固定阶段）

检验分两层：先基础检验确认理解，回答扎实则自动升级到面试官深挖。

**第一层：基础检验（2 问）**

答疑循环确认用户没问题后，说"好了，考你几个问题"，从以下六种中选 2 个。每次一问——先让用户回答，**用户答完后**再给出你的推荐答案做对照。

| # | 问法 | 测什么 |
|---|------|--------|
| 1 | "用你自己的话解释 X，不许用术语" | 是否内化 vs. 背定义 |
| 2 | "X 和 Y 的区别？给个具体场景"（Y 选一个容易混淆的相似概念） | 边界辨别 |
| 3 | "什么时候 X 不适用？或者用了反而更差？" | 对局限性的理解 |
| 4 | "给我一个实际会遇到的例子" | 抽象→具象的映射 |
| 5 | "关于 X，初学者最容易踩的坑是什么？你确定你没踩？" | 元认知 |
| 6 | "X 和你学过的 Z 有什么关联？"（Z 必须来自仓库已有笔记） | 知识整合 |

**回答扎实**：用自己的话、有例子、能说边界和局限、能和已有知识关联。
**回答有漏洞**：只是复述定义、给不出例子、对局限性说"好像都可以"。

**第二层：面试官深挖（基础检验扎实时自动触发）**

如果基础检验用户回答扎实，不要满足——你是面试官，你的目标是**找到他理解的边界**。

关键规则：
- **追问方向必须来自用户自己的回答，不是预设题库。** 用户说了什么，你就顺着那个点往下挖。
- **连续追问直到用户说"不知道"。** 那个"不知道"不是失败——它精准标定了下次学习的目标。
- **追问方式**：挑战用户自己的类比和表述。比如用户说"Attention 就像查字典"→ 追问"那如果字典里的词条会随着你查的动作实时变化呢？Self-Attention 是不是就不符合这个类比了？"
- **1-2 层即可**，不要无限挖（这不是真的面试）。触边就收。

```
用户："Attention 就像查字典，Query 是你要查的词，Key 是索引，Value 是释义。"
  ↓
面试官："好。但如果字典里的词条会根据你正在查的那个词动态变化呢？
        你要查'苹果'，字典里'手机'的释义就自动调整了——这还是查字典吗？
        Self-Attention 是不是打破了这个类比？"
  ↓
用户："……确实打破了。Self-Attention 里所有词同时互为 Query 和 Key，
      字典类比只适用于 Cross-Attention。"
  ↓
面试官："对。所以这个类比的边界在 Self-Attention 上失效——这恰恰是你真正理解
        Cross-Attention 和 Self-Attention 区别的关键。"
```

面试官的追问**暴露类比的边界、概念的例外、直觉的裂痕**——这些是比基础检验深一个层次的理解。

### 阶段三：补漏
- 基础检验有漏洞 → 针对性补充讲解，然后换方向再验证。
- 面试官深挖触到边界 → 确认边界位置，简短补充。补完后给出 2-3 个具体深入方向——"你可以继续的方向有：X 的数学推导、Y 的实现细节、Z 和其他概念的对比——还是先沉淀？"。
- 用户连续 2 个问题说"不知道" → 补充讲解后，同样给方向选项。
- 检验中发现的误区、用户的精辟比喻、知识的独特关联、**触到的理解边界**——这些是笔记最有价值的内容。

### 阶段四：沉淀

1. **判断是否值得沉淀**：对照"好笔记"标准——有类比、有误区、有自己的话（尤其是检验中暴露的理解，比搬运定义更有价值）。
2. 若值得 → 问用户"要沉淀成笔记吗？"。
3. 用户确认 → 读 `.agents/conventions.md`：
   若用户选择"继续深挖" → 回到阶段一，基于边界继续讲解（不需要重新走阶段零，目标不变）。

   **新建模式**：列 `wiki/` 判断主题（沿用或新建，新建须带 `_overview.md`）→ 新建 `<note>.md` → 更新 `_overview.md`。

   **更新模式**：读已有笔记 → 将新内容融入现有结构（新的类比加进「直觉/类比」，新的误区加进「常见误区」，新的关联加进「关联」）→ 刷新 `updated` 日期 → 如果新内容显著改变了笔记的核心理解，更新 `summary` → 更新 `_overview.md`（如果笔记的核心内容变了，_overview 的介绍也要同步）。

   **关键原则**：同一概念永远只有一篇笔记。两次学 Dropout 不是两篇笔记，是一篇笔记的两个版本。

4. **校验（必做）**：`wc -l <note.md>` 确认非空；确认 frontmatter 包含 `title`（概念名）、`topic`、`tags`、`summary`、`created`、`updated`；`git status` 确认改动符合预期；检查笔记内相对链接目标存在。
5. **提交**：`git add -A && git commit -m "learn <topic>: <一句话>"`。

## 什么是"值得沉淀"

**值得沉淀**（讲完主动问用户）：
- 用户表现出困惑后恍然大悟
- 内容有清晰的直觉类比和常见误区
- 填补了仓库现有知识的空白
- 检验中用户自己的精辟总结或独特理解角度
- 能够写出简洁的概念名标题、至少 2 个 tags、以及一个让 query 能扫描定位的 summary

**不值得沉淀**（讲完直接结束）：
- 用户只是随口一问，对话一笔带过
- 内容太浅（一句话就能说清楚的定义）
- 和已有笔记高度重复

## Gotchas（真实踩过的坑）

- **检验不是可选环节**，但检验之前必须先完成答疑循环——用户说"没问题了"才能进入检验。跳过答疑循环直接考，是最常见的错误。
- **沉淀不是默认终点，是用户主动选择**：补漏结束后问"继续深入还是先沉淀？"，给两个选项。不要问"可以沉淀了吗？"——这种问法把继续学习的路堵死了。如果用户选继续深挖，回到阶段一继续讲，不需要重新走阶段零。
- **永远不要假设用户准备好进入下一步**：每次确认都要给具体方向，不让用户面对空白。"有什么疑问？"——不够。"关于 X 的数学推导可以展开，Y 的实现细节也可以聊，还是都没问题进入检验？"——这才对。
- **面试官追问必须来自用户的回答**：不要用预设题库。用户说了什么，就顺着那个点深挖。预设问题无法触及用户独有的理解盲区。
- **一次只问一个问题**：禁止一次抛出多个问题——那是问卷，不是质询。每个问题等待用户回答后再继续。
- **推荐答案在用户回答之后给**：先出题让用户答，答完后再给推荐答案对照——不是出题时就附上。出题时附答案等于剧透。
- **不要未经确认就写文件**：讲完、考完、确认有价值后，仍要问"要沉淀成笔记吗？"，否则只聊天。
- **写入后必须校验文件非空**：曾发生并行批量写文件时单文件静默落盘失败（报成功但实际 0 行），靠 `wc -l` 才发现。
- **先查前置知识，不要假设用户懂**：新建模式时扫 wiki 判断用户是否有前置概念的基础。如果 wiki 里没有相关基础笔记，主动问——"X 依赖 Y，你之前学过吗？"用户说没学过就从 Y 开始教，不要硬讲 X。
- **讲解要精简，不要写教程**：控制在 3-5 个要点、~300 字。如果你写了表格、分段标题超过 3 个、代码示例超过 2 段——停下来，你已经超了。深度内容留到检验和补漏阶段，用户答不上来的时候再展开。
- **有公式必须写出来**：不能只说"用 softmax 归一化"而不给 softmax 公式。直觉让人理解，公式让人能用——两者缺一不可。
- **同一概念永远只有一篇笔记**：讲之前先查仓库。已有笔记 → 更新模式，不要新建重复笔记。两次学 Dropout 不是两篇笔记，是一篇笔记的两次迭代。
- **笔记存在 ≠ 用户掌握**：先读笔记正文看实际内容深浅——只有 TL;DR 的骨架 vs. 有类比有公式的完整笔记，判断完全不同。不要只看 frontmatter 就下结论。禁止自动重定向到 query——笔记存在用户也可能想继续深挖。
- **raw/ 不是知识的证明**：raw/ 里可能有下载的论文、拉取的网页，这些不是用户学过的标记。判断用户是否学过只能看 wiki/ 有没有笔记。不要把 raw/ 里的文件当成了用户的知识基础。

## 注意
