## TL;DR

`int cnt[26]{};` 全 0 不是因为"默认初始化"，而是因为空花括号 `{}` 触发了**值初始化（value initialization）**。值初始化对 `int` 这样的标量类型执行**零初始化（zero initialization）**——字节全置 0。对于 C 风格数组，中间还要经过聚合初始化的级联。静态存储期变量（global、static）即使不加 `{}` 也会在程序启动时零初始化——但那是另一条独立的触发路径。

## 核心概念

### 值初始化 vs 默认初始化

| 写法 | 初始化类别 | `int` 的结果 |
|------|-----------|-------------|
| `int x;` | 默认初始化 | 不确定值（脏数据） |
| `int x{};` | **值初始化** | 0 |
| `int x();` | **函数声明** | — |

C++ 标准规定，值初始化对非类、非数组的类型（标量类型）直接执行零初始化——把内存字节全部设为 0。

### 数组的级联链

C 风格数组是聚合类型。`int cnt[26]{}` 的执行路径：

```
列表初始化（看到 {}）
  → 花括号为空 + 目标是聚合类型
    → 聚合初始化
      → 每个元素没有显式指定初始值
        → 用 {} 拷贝初始化每个元素
          → 值初始化
            → int 是标量 → 零初始化
              → 0
```

三层机制串联，最终每个元素归零。

### 静态存储期的零初始化

```cpp
int cnt[26];          // 栈上 → 默认初始化 → 脏数据
static int cnt[26];   // 静态存储期 → 程序启动时零初始化 → 全 0
```

所有静态存储期变量（global、static、thread_local）在程序启动时先做一遍零初始化，再执行其他初始化。这和 `{}` 触发值初始化是两条独立的路径，但底层都是 zero-initialization。

### Most Vexing Parse

```cpp
int x();    // 声明了一个叫 x 的函数，返回 int！不是值初始化
int x{};    // 值初始化，x = 0
```

C++ 语法歧义规则：一切能被解析成函数声明的，优先解析成函数声明。这就是 C++11 力推 `{}` 统一初始化的历史原因——`{}` 永远不会被误解为函数声明。

### initializer_list 构造函数抢跑

```cpp
std::vector<int> v{5, 0};   // {5, 0}——两个元素
std::vector<int> v(5, 0);   // {0, 0, 0, 0, 0}——五个元素
```

当类型有 `std::initializer_list` 构造函数时，`{}` 优先匹配它。对 `int` 标量和 C 数组不存在这个问题，但对标准库容器是高频陷阱。

## 直觉 / 类比

`int cnt[26];` 和 `int cnt[26]{};` 就像租房子的两种方式：前者是上任租客的东西还在里面，后者是房东先做了一次深度保洁。空花括号 `{}` 就是那个保洁指令——把每个字节清零。

## 常见误区

- **"加 `{}` 就是初始化为 0"——不完整。** 对 `int` 标量和 C 数组确实变 0，但对有 `initializer_list` 构造的类，`{}` 调的是那个构造函数，含义完全不同。`std::vector<int> v{};` 是空 vector，不是装满 0 的 vector。
- **"默认初始化全 0"——概念错误。** `int cnt[26]{};` 是值初始化，不是默认初始化。默认初始化（`int cnt[26];`）对内置类型不做任何事，值是脏的。
- **"`int cnt[26] = {}` 也一样"——是 C 遗风。** C 语言里 `= {0}` 保证全 0，`= {}` 是 GNU 扩展。C++ 里 `int cnt[26]{};` 才是标准写法，语义等价但不依赖扩展。

## 面试常见问题

- **Q**: C++ 中 `{}` 和 `()` 初始化有什么区别？
  **A**: 三个关键区别：① `{}` 禁止窄化转换（`int x{3.14}` 编译错误，`int x(3.14)` 通过但截断）；② `{}` 可以用于聚合初始化和 `initializer_list` 构造（`vector<int> v{1,2,3}`），`()` 不行；③ `{}` 没有 Most Vexing Parse 问题——`Widget w();` 是函数声明，`Widget w{};` 才是构造对象。

- **Q**: `foo a{};` 和 `bar b{};` 的成员变量分别是什么值？（foo 的构造函数声明处 `= default`，bar 的定义处 `= default`）
  **A**: `a.a` 是 0，`b.b` 是不确定值。关键区别：foo 的 `= default` 在声明处 → 不是 user-provided 构造函数 → 值初始化时先做零初始化 → 成员归零。bar 的 `= default` 只在定义处 → 是 user-provided → 值初始化只调用默认构造，不做零初始化 → 成员未初始化。`= default` 的位置直接影响初始化行为，这是 C++ 标准里最隐蔽的规则之一。

- **Q**: 默认初始化、值初始化、零初始化分别是什么？
  **A**: **默认初始化**：类类型调默认构造，内置类型不做任何事（值不确定）。**值初始化**：类类型调默认构造（但如果默认构造不是 user-provided，先零初始化），内置类型零初始化。**零初始化**：把内存字节全置 0，在程序启动时应用于所有静态存储期变量，也是值初始化的底层步骤。简单记忆——默认初始化不保证安全，值初始化保证安全，零初始化是底层机制。

- **Q**: 声明一个 1 万长度的数组 `int arr[10000] = {0, 0, 0, ... 只给了前 135 个}`，剩余 9865 个元素的值是什么？这么大的数据是怎么初始化的？
  **A**: 剩余元素全是 0。机制：这是聚合初始化——显式给出了前 135 个值，剩余元素用 `{}` 做值初始化（对 `int` 就是零初始化）。对于全局/静态大数组，BSS 段在程序加载时由操作系统批量清零（`memset` 或类似机制），不占磁盘空间。对于栈上的局部大数组，编译器生成循环或 `memset` 调用批量置零。

- **Q**: 初始化为 0 的全局变量放在 BSS 段还是 DATA 段？
  **A**: BSS 段。BSS（Block Started by Symbol）存放未初始化或初始化为 0 的全局/静态变量。非零初始化的全局变量放在 DATA 段。BSS 不占可执行文件磁盘空间——只需记录起始地址和长度，加载时 OS 批量清零。

- **Q**: `std::vector<int> v{5, 0};` 和 `std::vector<int> v(5, 0);` 各得到什么？
  **A**: `v{5, 0}` 得到包含两个元素 `{5, 0}` 的 vector（`initializer_list` 构造），`v(5, 0)` 得到包含 5 个元素、每个都是 0 的 vector（fill 构造）。`{}` 优先匹配 `initializer_list` 构造函数——C++ 初始化最高频的陷阱。

- **Q**: 未初始化变量会有什么后果？
  **A**: 读取未初始化的内置类型变量是**未定义行为（UB）**。局部 `int x;` 的值是栈上残留数据，可能每次运行不同——调试时偶尔"碰巧"是 0 让人误以为正确，发布后随机出错。C++ 核心准则 ES.20："始终初始化变量"。
