## TL;DR

给定 `height[i]` 表示柱状图每列高度，计算下雨后柱子之间能积多少水。
**核心公式**：`water[i] = max(0, min(左边最高, 右边最高) - height[i])`。
双指针维护 `pre_max`（左侧最高）和 `suf_max`（右侧最高），每次移动较矮的那侧——
因为矮侧的水量已经确定（高侧有更强的墙兜底）。时间 O(n)，空间 O(1)。
思维关键是**问题分解**：把"整个图能接多少水"拆成"每根柱子上面能接多少水"。

## 核心概念

### 问题分解：局部化（Localize）

拿到全局问题"整个柱状图能接多少水"时，不要试图一次性算出答案。
换一个问法：**每根柱子 `i` 上面能接多少水？**

```
柱子高度：[2, 0, 1]
图形：     █
           █   █
           █ _ █    ← 水在 index 1 上面
```

盯着 index 1 那根高度为 0 的柱子：
- 往左看，最高柱子是 2
- 往右看，最高柱子是 1
- 水不能超过左边的墙，也不能超过右边的墙 → 水位 = min(2, 1) = 1
- 柱子本身是 0 → 水量 = 1 − 0 = 1

**每个位置的水量完全由左右两堵最高的墙决定，与其他几百根柱子无关。**
算出每个位置的水量，加起来就是答案。这叫"局部化"——把全局问题拆成独立子问题。

### 推导链条

**Level 0：暴力（每个位置独立算）**

对每个 `i`，向左扫描找 `left_max`，向右扫描找 `right_max`，计算 `water[i]`。O(n²)。

**Level 1：预计算数组**

暴力的问题是每个 `i` 都重新扫描左右。用两个数组缓存：
- `left_max[i]`：从左到右扫描一次
- `right_max[i]`：从右到左扫描一次
- 第三遍扫描算答案

O(n) 时间，O(n) 空间。这步没有智力跳跃——有暴力解法，优化方向是自然的。

**Level 2：双指针**

盯着 `left_max[i]` 和 `right_max[i]` 两个数组看，关键观察是：

> 算 `water[i]` 时只关心 `min(left_max[i], right_max[i])`——**谁小谁决定水量**。
> 大的那侧的值只要 ≥ 小的那侧就行，具体值不重要。

引入双指针 `left = 0`，`right = n-1`，维护 `pre_max` 和 `suf_max`：

- 若 `pre_max < suf_max`：对 `left` 位置，左边最高确认为 `pre_max`，
  右边最高**至少是** `suf_max`（可能更高，但 min 不会变小），
  因此 `water[left] = pre_max - height[left]` 已确定 → `left++`
- 若 `pre_max >= suf_max`：对称，`right--`

```cpp
int trap(vector<int>& height) {
    int ans = 0;
    int pre_max = 0, suf_max = 0;
    int left = 0, right = height.size() - 1;
    while (left < right) {
        pre_max = max(pre_max, height[left]);
        suf_max = max(suf_max, height[right]);
        if (pre_max < suf_max) {
            ans += pre_max - height[left];
            left++;
        } else {
            ans += suf_max - height[right];
            right--;
        }
    }
    return ans;
}
```

## 直觉 / 类比

想象从两端往中间收网。你手里有两根"水位尺"——左边记录见过的最高柱子，
右边记录从右往左见过的最高柱子。矮的那一侧决定了当前位置最多能存多高的水，
因为另一侧有更高的墙兜底，瓶颈在矮侧。所以放心地在矮侧结算，然后收窄范围。

更直白：两个人从两端往中间走，矮的那个人先倒水——他站的地方水位已经确定，
因为对面有个更高的人堵着，水不会从那边流走。

## 常见误区

- **以为水量 = `pre_max - height[i]`（忽略右侧）。**
  这只在右侧有更高墙时成立。如果右侧全比当前矮，水根本存不住。
  双指针的正确性来自"矮的那侧做 min"——只有确认了对面有更高的兜底墙，
  才能用当前侧的最大值结算。

- **以为移动的是高的一侧。**
  恰好相反。高的一侧还有可能继续升高或降低，水量不确定；
  矮的一侧已经确定，所以先结算它。

- **试图用"找凹槽"的整体思路解。**
  看到水就想到盆地和容器，试图找成对的左右边界——越找越乱。
  正确的起点是"每根柱子独立看"。

## 面试常见问题

- **Q**: 面试官问为什么可以安全地移动较矮一侧的指针？
 **A**: 因为决定 `water[i]` 的是 `min(left_max, right_max)`，矮的那侧就是当前有效的上限。
 当 `height[left] <= height[right]` 时，右侧至少有一个比 `height[left]` 高的墙（即 `height[right]`），
 所以 `max_left` 就是当前左侧的约束。右侧的确切最大值不需要知道——反正比左侧高。
 同理对称。关键：**比较的是 `height[left]` 和 `height[right]`，不是 `max_left` 和 `max_right`。**
 （但如果比较的是 pre_max/suf_max 也没错，两种写法等效——只是直觉不同。）

- **Q**: 为什么双指针不需要 `max(0, ...)` 来做 clamp？
 **A**: 因为 `pre_max` 已经包含了当前柱子的高度（`pre_max = max(pre_max, height[left])`），
 所以 `pre_max - height[left]` 一定 >= 0。不需要额外保护。

- **Q**: 如果数组长度小于 3，或者全是 0 怎么办？
 **A**: 数组长度 < 3 不可能形成山谷，直接返回 0。全是 0 的情况公式自然处理：water = min(0, 0) - 0 = 0。
 可以在开头加一个 `if (n < 3) return 0;` 的 guard clause，但不加也可以——循环自然跳过。

- **Q**: 暴力解法怎么一步步优化到双指针？
 **A**: 面试官期待的推导链条——
 1. 暴力：每个位置左右扫描，`water[i] = min(左侧最高, 右侧最高) - height[i]`。O(n²)。
 2. 预计算数组：把左右最大值缓存到 `left_max[]` 和 `right_max[]`，三遍扫描。O(n) 时间，O(n) 空间。
 3. 双指针：观察到 `left_max[i]` 单调递增、`right_max[i]` 单调递减，可以用两个指针从两端向中间收敛，
    维护运行最大值替代数组。O(n) 时间，O(1) 空间。

- **Q**: 能不能用单调栈解接雨水？和双指针比有什么优劣？
 **A**: 可以。单调递减栈：遇到更高的柱子时，弹出栈顶形成一个凹槽，左右边界分别是栈的下一个元素和当前柱。
 单调栈更直观（模拟真实的接水过程），但需要 O(n) 空间。双指针更省空间。
 面试中两种都提到会加分——表明你理解多种思路。实际中双指针更常用。

- **Q**: 面试官问这道题你最开始是怎么想到解法的？
 **A**: 此题的核心起点不是双指针，而是**问题分解**——把整个图能接多少水拆成每根柱子能接多少水。
 一旦写出 `water[i] = min(left_max[i], right_max[i]) - height[i]`，剩下的就是优化这个公式的计算方式。
 面试官真正想看的：你能不能从几何直觉出发（水被两边的墙挡住），独立推导出核心公式。

- **Q**: 什么输入数据会让双指针解法出 bug？
 **A**: 单调递增序列（如 `[1,2,3,4,5]`）：每个位置的左边最高都比右边最高小，答案应该是 0——
 双指针也应该返回 0（因为 `pre_max` 始终等于当前高度，`pre_max - height[left] = 0`）。
 单调递减序列同理。但如果把条件写成 `pre_max < suf_max` 而忘了先更新 `pre_max/suf_max`，
 可能在某些边界出问题——先 update max，再算水，这个顺序不能颠倒。
