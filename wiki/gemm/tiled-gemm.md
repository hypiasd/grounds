## TL;DR

分块 GEMM = **输出块网格 × 每块的 K 循环累加**，与 Triton kernel body 同构。`BLOCK_M/N/K` 同时决定并行度（块数）与每程序工作量（K 循环），是 autotune 的核心旋钮。分块"能算对"靠加法的可加性，"算得快"靠数据复用。

## 核心概念

### 1. 两个可加性（合法性来源）
- **输出元素独立**：`C[i,j] = Σ_k A[i,k]·B[k,j]` 只依赖 A 的第 i 行 + B 的第 j 列，C 的块之间无任何数据依赖 → 可任意切片、可并行。
- **K 维求和可拆**：一个 C 块 = `Σ_{kk} A[I,kk:kk+BK]·B[kk:kk+BK,J]`，K 维只是普通加法（交换/结合律成立）→ 可分段、可乱序、可并行累加。

分块"能算对"是因为加法可分组；分块"算得快"是因为数据复用。

### 2. 怎么切 A / B
- A 被切两维：**M（行块，外层 bi）+ K（K段，内层 bk）**。
- B 被切两维：**K（K段，内层 bk）+ N（列块，外层 bj）**。
- `A_tile = A[bi:i_end, bk:k_end]` 与 `B_tile = B[bk:k_end, bj:j_end]` 的 **K 段范围必须相同**才能相乘（形状 `[BM,BK]×[BK,BN]`）。不是"先切 A 行、再切 B 列、再各自切 K"的串行，而是外层 `(bi,bj)` 同时锁定行块/列块，内层 `bk` 滑动时同步取对齐 K 段。

### 3. 写回用 `=` 而非 `+=`
- 块间不重叠：外层按 M/N 切，每个 `(bi,bj)` 块对应 C 唯一子矩阵，无两程序写同址。
- K 累加已在局部 `acc` 内完成（`acc += A_tile @ B_tile` 是矩阵级逐 K 段累加，K 维全吃完才写回）。
- 真需 `+=`（GPU 上 `tl.atomic_add`）仅 split-K 场景：把 K 维也分给多程序、各算某块 C 的"一段 K"需合并到同址。

### 4. 三张示意图
- **图1 输出块网格**：C 按 M/N 切，每块一个程序，互不重叠（网格 = ⌈M/BM⌉ × ⌈N/BN⌉）。
- **图2 单块数据流**：`A 行块[BM,K] × B 列块[K,BN] = C 块[BM,BN]`。
- **图3 K 维分段累加**：长条切 BK 小段，逐段搬入乘、攒进 acc，K 吃完写回 `=`。

## 直觉 / 类比

算全班总分 = 把每个人分数相加。你可以**按座位（前排/后排）分块求和再加**，也可以**按科目段（数学段/语文段）分了再加**——加法分组不改变总和。分块 GEMM 就是"按座位块分、按科目段加"。

## 常见误区

- **误区：A 只按行切、B 只按列切。** 实际 A、B 各被切了两个维度，且 A_tile/B_tile 必须共享同一段 K 才能相乘。
- **误区：写回该用 `+=`。** 块间不重叠 + K 累加已在 acc 完成，写回是最终值，用 `=`；`+=` 只在 split-K 合并同址时需要（GPU 上很贵）。
- **误区：分块只是为了让结果算对。** 分块"能算对"是加法属性，但所有 GEMM kernel 都分块是因为分块能在带宽墙上活下来（见 [HBM 流量与数据复用](../gpu/hbm-traffic.md)）。

## 面试常见问题

- Q：为什么 GEMM 一定要分块？
  A：不是必须分块才能算对，而是分块把"加法的可加性"翻译成"数据的复用率"，才能在固定 HBM 带宽下逼近算力墙。
- Q：Triton matmul 里写回为什么是 `=` 不是 `atomic_add`？
  A：官方教程单程序吃完整 K，块间不重叠、K 累加已在寄存器 acc 完成，`=` 即正确语义；`atomic_add` 只在 split-K 多程序合并同址时才需要。
