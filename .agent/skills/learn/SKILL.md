# learn

## 何时用（触发）
- 用户问一个新知识 / 概念 / 原理。
- 用户说："讲讲 X"、"X 是什么"、"帮我理解 Y"、"用老师模式讲一下"。
- 用户问的内容**在仓库现有笔记中找不到**（如果是已有知识，走 query）。

## 目标（完成时仓库应处于的状态）
- 用户已理解该概念（直觉 + 形式化 + 误区都覆盖）。
- 若用户认可沉淀：仓库新增/更新一篇 `wiki/<topic>/<note>.md`，对应 `_overview.md` 已同步，且已 `git commit`。

## 流程
1. **老师模式讲解**（先对话，不急着写文件）
   - 先给**直觉 / 类比**，再给形式化细节；主动指出**常见误区**。
   - 不确定处明确说"我不确定"，不要编造。
2. **判断是否沉淀**：讲解结束后，若内容有价值（满足"好笔记"标准——有类比、有误区、有自己的话），问用户"要沉淀成笔记吗？"。
3. **判断主题**：列 `wiki/` 看已有主题，按内容归属；无合适主题 → 新建目录并按 `conventions.md` 模板建 `_overview.md`。
4. **沉淀**（仅当用户认可）：在 `wiki/<topic>/` 下新建 `<note>.md`（读 `.agent/conventions.md` 后按模板写）；更新 `_overview.md`。
5. **校验（必做）**：`wc -l <note.md>` 确认非空；`git status` 确认改动符合预期；检查笔记内相对链接目标存在。
6. **提交**：`git add -A && git commit -m "learn <topic>: <一句话>"`。

## 什么是"值得沉淀"

**值得沉淀**（讲完主动问用户）：
- 用户表现出困惑后恍然大悟（"哦原来是这样"）
- 内容有清晰的直觉类比和常见误区
- 填补了仓库现有知识的空白

**不值得沉淀**（讲完直接结束）：
- 用户只是随口一问，对话一笔带过
- 内容太浅（一句话就能说清楚的定义）
- 和已有笔记高度重复

## Gotchas（真实踩过的坑）

- **不要未经确认就写文件**：先讲，结尾问"要沉淀成笔记吗？"，否则只聊天。因为用户可能只是临时想了解，不一定想存。
- **写入后必须校验文件非空**：曾发生并行批量写文件时单文件静默落盘失败（报成功但实际 0 行），靠 `wc -l` 才发现。
- **主题名用小写中划线**，且不要预建一堆空目录——主题由内容自然生长。
- **不要重复写 Claude 已懂的通用知识**：只补项目特有约定（见 AGENTS.md / conventions.md）。

## 注意
- 保持和现有笔记的互链，避免孤儿页。
- 关联：`AGENTS.md`、`.agent/conventions.md`
