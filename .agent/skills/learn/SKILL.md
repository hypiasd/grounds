# learn

## 何时用（触发）
- 用户问一个新知识 / 概念 / 原理。
- 用户说："讲讲 X"、"X 是什么"、"帮我理解 Y"。
- 用户问的内容**在仓库现有笔记中找不到**（如果是已有知识，走 query）。

## 目标（完成时仓库应处于的状态）
- 用户已理解该概念（直觉 + 形式化 + 误区都覆盖），**且经过主动检验确认理解扎实**。
- 若用户认可沉淀：仓库新增/更新一篇 `wiki/<topic>/<note>.md`，对应 `_overview.md` 已同步，且已 `git commit`。

## 流程

learn 分四个阶段：**讲解 → 检验 → 补漏 → 沉淀**。检验是固定阶段，不需要用户说"考考我"。

### 阶段一：讲解
1. 先给**直觉 / 类比**，再给形式化细节。
2. 主动指出**常见误区**。
3. 不确定处明确说"我不确定"，不要编造。

### 阶段二：检验（AI 主动发起，固定阶段）

讲完后，直接说"好了，考你几个问题"，然后从以下六种 Forcing Question 中选 2-3 个——**每次只问一个，等用户回答后再问下一个，每个问题附带你的推荐答案**：

| # | 问法 | 测什么 |
|---|------|--------|
| 1 | "用你自己的话解释 X，不许用术语" | 是否内化 vs. 背定义 |
| 2 | "X 和 Y 的区别？给个具体场景"（Y 选一个容易混淆的相似概念） | 边界辨别 |
| 3 | "什么时候 X 不适用？或者用了反而更差？" | 对局限性的理解 |
| 4 | "给我一个实际会遇到的例子" | 抽象→具象的映射 |
| 5 | "关于 X，初学者最容易踩的坑是什么？你确定你没踩？" | 元认知 |
| 6 | "X 和你学过的 Z 有什么关联？"（Z 必须来自仓库已有笔记） | 知识整合 |

**回答扎实**：用自己的话、有例子、能说边界和局限、能和已有知识关联。
**回答有漏洞**：只是复述定义、给不出例子、对局限性说"好像都可以"。

### 阶段三：补漏
- 用户全部回答扎实 → 简短确认，进入沉淀阶段。
- 某个回答暴露漏洞 → 针对性补充讲解，然后换一个方向再问 1 个问题验证。
- 用户连续 2 个问题说"不知道" → 不追问，直接补充讲解，然后进入沉淀（检验暴露的盲区本身就是笔记的素材）。
- 检验过程中发现的误区、用户的精辟比喻、知识的独特关联——这些是笔记最有价值的内容。

### 阶段四：沉淀
1. **判断是否值得沉淀**：对照"好笔记"标准——有类比、有误区、有自己的话（尤其是检验中暴露的理解，比搬运定义更有价值）。
2. 若值得 → 问用户"要沉淀成笔记吗？"。
3. 用户确认 → 列 `wiki/` 判断主题（沿用或新建，新建须带 `_overview.md`）→ 读 `.agent/conventions.md` → 写 `<note>.md` → 更新 `_overview.md`。
4. **校验（必做）**：`wc -l <note.md>` 确认非空；`git status` 确认改动符合预期；检查笔记内相对链接目标存在。
5. **提交**：`git add -A && git commit -m "learn <topic>: <一句话>"`。

## 什么是"值得沉淀"

**值得沉淀**（讲完主动问用户）：
- 用户表现出困惑后恍然大悟
- 内容有清晰的直觉类比和常见误区
- 填补了仓库现有知识的空白
- 检验中用户自己的精辟总结或独特理解角度

**不值得沉淀**（讲完直接结束）：
- 用户只是随口一问，对话一笔带过
- 内容太浅（一句话就能说清楚的定义）
- 和已有笔记高度重复

## Gotchas（真实踩过的坑）

- **检验不是可选环节**：讲完必须主动发起检验，不需要等用户说"考考我"。用户说"不知道"时追问"那你觉得可能是什么？"，不要直接开讲——先逼思考，再补讲解。
- **一次只问一个问题**：禁止一次抛出多个问题——那是问卷，不是质询。每个问题等待用户回答后再继续。
- **每个问题附带推荐答案**：示范你期望的回答深度，同时暴露你的判断偏差。
- **不要未经确认就写文件**：讲完、考完、确认有价值后，仍要问"要沉淀成笔记吗？"，否则只聊天。
- **写入后必须校验文件非空**：曾发生并行批量写文件时单文件静默落盘失败（报成功但实际 0 行），靠 `wc -l` 才发现。
- **主题名用小写中划线**，且不要预建一堆空目录——主题由内容自然生长。

## 注意
- 保持和现有笔记的互链，避免孤儿页。
- 关联：`AGENTS.md`、`.agent/conventions.md`
