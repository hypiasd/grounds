# grill

## 何时用（触发）
- 用户说："grill me on X"、"考考我对 X 的理解"、"质询一下"、"问问我"、"来考我"。
- 用户学完一个概念后，想主动检验自己到底懂没懂。
- 注意：grill 是检验已有理解，不是教新东西（那是 learn），也不是查笔记作答（那是 query）。

## 目标（完成时仓库应处于的状态）
- 用户对自己理解的深度和盲区有清晰认知。
- 若过程中发现值得沉淀的洞见（误区、类比、关联），提示用户走 capture。
- grill 本身不写文件。

## 核心机制

### 一次只问一个问题

等待用户回答后再问下一个。禁止一次抛出多个问题——那是问卷，不是质询。

### 每个问题附带推荐答案

格式：**"推荐答案：X，因为 Y"**。这有几个作用：
- 示范你期望的回答深度
- 暴露你的判断偏差（用户说"不对，我觉得是 Z"比"你说呢？"更有信息量）
- 防止无限兜圈

### 深度优先，走通一个分支再换

如果用户的回答暴露了某个薄弱点，顺着追问，不要跳到新话题。把一条线问透了再换方向。

### 能查笔记就不问

如果问题可以通过读 `wiki/` 中的已有笔记回答——直接读，然后基于笔记内容追问，而不是凭空提问。

## 六种 Forcing Questions（学习版）

根据用户的回答动态选择，不按固定顺序。每次只用一个。

### 1. "用你自己的话解释 X，不许用术语"
**逼出什么**：是否真的内化了，还是在背定义。
**反例（软问题）**："你理解 X 吗？"
**推荐答案示例**："注意力机制就是——模型看一句话的时候，不是每个词都同样重要，它会'盯着'和当前任务最相关的几个词。就像你在嘈杂的咖啡馆听朋友说话，会自动忽略背景噪音。"

### 2. "X 和 Y 的区别是什么？给个具体场景"
**逼出什么**：边界辨别能力。Y 选一个容易混淆的概念。
**反例（软问题）**："你知道 X 和 Y 不一样吧？"
**推荐答案示例**："Self-Attention 和 Cross-Attention 的区别：Self-Attention 是句子内部词和词之间互相关注——'我'看'苹果'；Cross-Attention 是跨两个序列——翻译时，解码器的每个词去关注编码器的所有词。"

### 3. "什么时候 X 不适用？或者用了反而更差？"
**逼出什么**：对局限性的理解。只知"什么时候用"是半懂。
**反例（软问题）**："X 有什么缺点？"
**推荐答案示例**："Attention 在短序列上优势不明显——如果句子只有 3-5 个词，RNN 可能更快且效果差不多。另外 Attention 的 O(n²) 复杂度在超长序列上是瓶颈。"

### 4. "给我一个你实际会遇到的具体例子"
**逼出什么**：从抽象到具象的映射能力。泛泛而谈说明没真懂。
**反例（软问题）**："你能举个例子吗？"

### 5. "关于 X，初学者最容易踩的坑是什么？你确定你没踩进去？"
**逼出什么**：元认知——知道自己不知道什么。能说出误区说明已经过了那个阶段。
**反例（软问题）**："你觉得自己学得怎么样？"
**推荐答案示例**："最大的坑是以为 Multi-Head Attention 就是多做几次取平均——实际上每个 Head 在不同的投影子空间工作，拼接后才是完整信息。我确定我没踩，因为我知道每个 Head 的 W_Q/W_K/W_V 矩阵是独立训练的。"

### 6. "X 和你之前学过的 Z 有什么关联？"
**逼出什么**：知识整合能力。孤立的知识点容易遗忘。
**反例（软问题）**："你还学过相关的吗？"
**注意**：Z 应该是仓库中已有笔记覆盖的概念——先读 `wiki/` 确认，不要凭空问一个不存在的。

## 流程

1. **确定目标**：用户说"grill me on X"→ 确认 X 是什么。如果仓库有相关笔记，先读一遍做功课。
2. **开场**：简要说明规则——"我会一个问题一个问题地问，每个问题附带我的推荐答案。你回答后我会根据你的回答决定追问还是换方向。准备好了就开始。"
3. **质询循环**：
   - 从 6 种 Forcing Question 中选一个最合适的
   - 附带推荐答案
   - 等待用户回答
   - 判断：回答扎实 → 换方向；回答模糊/有漏洞 → 深挖；回答暴露严重误解 → 标记，问完当前线后切到澄清模式
4. **收尾**：
   - 总结：指出用户回答中最扎实的部分和最薄弱的部分
   - 如果暴露了值得沉淀的洞见 → 提示"这个可以走 capture 记下来"
   - 如果暴露了严重的知识空白 → 提示"这个建议走 learn 补一下"

## 什么算"回答扎实"vs."回答有漏洞"

**扎实**：
- 用了自己的话而不是背定义
- 给了具体的例子
- 能说出边界和局限
- 能和已有知识建立关联

**有漏洞**：
- 只是换了个说法复述定义
- 给不出例子，或者说"大概就是那样"
- 对"什么时候不适用"的回答是"好像都可以用"
- 无法和任何已有知识建立关联

## 何时停止

- 3 个方向用户都回答扎实 → 收尾，给正面反馈
- 用户连续 2 个问题说"不知道" → 收尾，建议走 learn
- 用户主动说"够了" → 收尾
- 最多问 8 个问题（防止马拉松）

## Gotchas（真实踩过的坑）

- **不要变成讲课**：用户说"不知道"时，追问"那你觉得可能是什么？"而不是直接开讲。直接开讲就把 grill 变成了 learn，失去了检验的意义。
- **不要用软问题凑数**："你觉得自己理解了吗？""还有疑问吗？"——这些不提供任何信号。每个问题必须是 Forcing Question。
- **推荐答案不是标准答案**：推荐答案是锚点，用户说"不对，我觉得是 X"比你复述你的推荐答案更有价值。不要纠正用户来匹配你的推荐答案——而是通过追问测试他的答案是否站得住。
- **仓库有笔记时必读**：grill 之前先读相关 wiki 笔记，避免问出笔记里已经写得很清楚的问题（会显得你没做功课），也避免遗漏笔记里标注的已知误区。
- **不要在用户明显沮丧时继续追问**：grill 的目的是暴露盲区，不是打击信心。如果用户连续"不知道"，切换到建设性模式——"没关系，这说明这个方向值得学，要不要走 learn？"

## 注意
- grill 不写文件；若过程中产生值得沉淀的洞见，提示用户走 capture。
- grill 和 learn 互补：learn 是"你讲我听"，grill 是"我讲你问"。同一概念可以学完后立刻 gril——learn → grill → capture（理想的完整学习闭环）。
